# 03 GRPO 奖励函数设计

GRPO 的核心不是“多训几步”，而是 reward function。奖励函数写错，模型会认真学错。

本阶段使用一个故意简单的规则奖励函数：

```text
scripts/stage3_reward_plugin.py
```

它不是生产级奖励模型，作用是让你能逐项解释：

1. 哪些回答被奖励。
2. 哪些回答被惩罚。
3. reward 设计会带来什么副作用。

## 1. GRPO 数据和 reward 的关系

GRPO 数据文件：

```text
data/stage3_grpo_prompts.jsonl
```

一条样本类似：

```json
{
  "id": "grpo_a800_safety",
  "messages": [
    {"role": "system", "content": "你是 A800 Qwen 微调学习教练..."},
    {"role": "user", "content": "给出共享 A800 上启动训练前的安全检查清单。"}
  ],
  "required": ["nvidia-smi", "显存", "CUDA_VISIBLE_DEVICES", "日志"],
  "forbidden": ["直接停止", "随便杀", "不用检查"],
  "rubric": "奖励安全检查、GPU 限定和日志意识。"
}
```

字段解释：

| 字段 | 作用 |
|---|---|
| `messages` | 给模型生成回答的 prompt |
| `required` | 希望回答里覆盖的关键概念 |
| `forbidden` | 出现后要扣分的危险表达 |
| `rubric` | 给人看的评分标准 |

`ms-swift` 的 GRPO 会把这些额外字段传给自定义 ORM，所以 reward function 可以读取 `required` 和 `forbidden`。

## 2. 奖励函数入口

核心类：

```python
class Stage3QualityORM(ORM):
    def __call__(self, completions, required=None, forbidden=None, rubric=None, **kwargs):
        ...
```

你要理解：

| 参数 | 含义 |
|---|---|
| `completions` | 模型对同一批 prompt 生成的回答 |
| `required` | 数据里传入的必备关键词 |
| `forbidden` | 数据里传入的禁止表达 |
| `rubric` | 当前脚本不直接打分，但保留给人工复盘 |
| `**kwargs` | 其它额外字段，方便以后扩展 |

注册 reward：

```python
orms["stage3_quality"] = Stage3QualityORM
```

训练时通过下面参数使用：

```bash
--external_plugins scripts/stage3_reward_plugin.py
--reward_funcs stage3_quality
```

## 3. 当前 reward 怎么打分

当前得分由 4 类组成：

| 维度 | 最高分 | 解释 |
|---|---:|---|
| required 覆盖 | `0.45` | 必备关键词出现越多，分越高 |
| 结构化表达 | `0.20` | 分点、步骤、清单更高 |
| 长度合适 | `0.15` | 太短或太长都不满分 |
| 安全/验证意识 | `0.20` | 出现风险、检查、验证、记录等词加分 |
| forbidden 惩罚 | `-0.35` | 出现危险表达扣分 |

最终分数会限制在 `[0, 1]`。

## 4. 为什么不能只奖励关键词

只奖励关键词会导致模型学会这种坏行为：

```text
nvidia-smi 显存 CUDA_VISIBLE_DEVICES 日志。
```

这句话关键词全有，但没有解释、没有步骤、没有风险边界。

所以当前 reward 还加入了：

1. 结构化表达。
2. 长度区间。
3. 安全/验证意识。
4. forbidden 惩罚。

但你也要清楚：这仍然很粗糙。

## 5. 当前 reward 的已知缺陷

| 缺陷 | 后果 |
|---|---|
| 关键词匹配很浅 | 模型可能堆关键词 |
| 不理解语义等价 | “查看 GPU 占用”不一定匹配 `nvidia-smi` |
| 不验证事实 | 错误但带关键词的回答可能拿高分 |
| 不看上下文完整性 | 回答漏掉关键步骤时未必严惩 |
| 不区分严重错误 | 小格式问题和危险操作可能都只扣固定分 |

这正是你要学习的地方：GRPO 的难点不是调用框架，而是 reward 设计和失败归因。

## 6. Reward 自检

本地同步到远端后运行：

```powershell
.\scripts\a800\run_stage3_reward_selftest.ps1
```

它会在远端执行：

```bash
python3 scripts/stage3_reward_plugin.py
```

预期输出类似：

```json
{"scores": [1.0, 0.0]}
```

你要确认：

1. 好答案得分高于坏答案。
2. forbidden 命中会扣分。
3. 没有 `ms-swift` 环境时，本地也能做基础自检。

## 7. 你必须能回答的问题

学完这份文档后，你要能回答：

1. 为什么 `required` 不能太多？
2. 为什么 `forbidden` 不能写得太泛？
3. 为什么结构化奖励可能让模型变模板化？
4. 为什么长度奖励可能鼓励废话？
5. 如果模型开始堆关键词，下一轮 reward 怎么改？
6. 如果模型回答正确但没出现指定词，应该改数据还是改 reward？

下面是这 6 个问题的合格回答。重点不是背 reward 公式，而是能解释 reward 为什么会被模型“钻空子”。

### 7.1 为什么 `required` 不能太多？

一句话回答：

> `required` 太多会把 reward 从“奖励好回答”变成“奖励关键词覆盖率”，模型容易为了拿分硬塞关键词。

在当前脚本里，`required` 的打分逻辑是：

```python
matched = sum(1 for term in required_terms if term.lower() in text.lower())
score += 0.45 * matched / len(required_terms)
```

这意味着 `required` 越多，模型越像在做“关键词填空”。如果一个问题只需要解释 `top_p` 和 `temperature`，但你要求它同时出现 `softmax`、`logits`、`nucleus sampling`、`概率分布`、`候选集合`、`重复惩罚`，模型就可能把这些词都堆进答案里，反而牺牲清晰度。

更严重的是，如果 `required` 太多且有些词不是每个正确答案都必须出现，那么正确但表达自然的答案会被低估。

合理做法：

| 场景 | `required` 建议 |
|---|---|
| 概念解释 | 只放 2 到 4 个核心概念 |
| 操作清单 | 放必须执行的关键检查项 |
| 安全规则 | 放不可缺少的风险边界 |
| 同义表达很多 | 不要全塞进 `required`，改成同义词组或语义判断 |

对你当前项目来说，`required` 应该奖励“答案必须覆盖的主干”，不要把所有你希望看到的词都写进去。

### 7.2 为什么 `forbidden` 不能写得太泛？

一句话回答：

> `forbidden` 太泛会误伤正常回答，让模型为了避开扣分而不敢讲关键风险。

当前脚本的 `forbidden` 逻辑是只要命中任意禁止词就扣分：

```python
if _contains_any(text, forbidden_terms):
    score -= 0.35
```

这是一种很硬的规则。比如你把 `停止` 写进 `forbidden`，那么下面两个回答都会被扣分：

| 回答 | 是否应该扣分 |
|---|---|
| “不要直接停止别人的 vLLM 服务。” | 不应该扣分，这是安全提醒 |
| “直接停止其他服务再训练。” | 应该扣分，这是危险操作 |

所以 `forbidden` 不能只写泛词，要写具体危险表达。比如不要写 `停止`，而要写：

1. `直接停止`
2. `随便杀`
3. `不用检查`
4. `不看 nvidia-smi`
5. `停掉别人服务`

更好的做法是区分“否定语境”和“肯定危险操作”。例如“不要直接停止”不该扣分，“直接停止其他服务”才该扣分。当前 v1 脚本还没有做到这一点，所以 `forbidden` 必须写得窄。

### 7.3 为什么结构化奖励可能让模型变模板化？

一句话回答：

> 结构化奖励会鼓励分点、步骤、清单，但如果权重太高，模型会学会固定套模板，而不是根据问题组织答案。

当前 `_structure_score` 会奖励这些形式：

1. 多个 bullet。
2. 多行内容。
3. 出现“第一、第二、首先、然后、最后、步骤、清单”等过渡词。

这对操作类问题是有用的。例如“启动训练前检查什么”确实适合清单。

但如果所有问题都奖励这种结构，模型可能变成：

```text
第一，...
第二，...
第三，...
最后，...
```

即使用户只问一个很短的概念问题，它也强行写成固定步骤。这就是模板化。

判断标准：

| 问题类型 | 是否应该强奖励结构 |
|---|---|
| 操作流程、排错、训练清单 | 应该 |
| 概念解释 | 轻度奖励即可 |
| 一句话判断 | 不应该强奖励 |
| 对比分析 | 可以奖励表格或分点，但不能固定格式 |

下一轮可以把结构奖励改成“按任务类型启用”。例如数据里新增 `task_type` 字段：`checklist` 才强奖励结构，`concept` 只轻度奖励清晰表达。

### 7.4 为什么长度奖励可能鼓励废话？

一句话回答：

> 长度奖励如果只看字数区间，模型会为了进入高分区间补无效内容。

当前长度逻辑是：

```python
if 80 <= length <= 700:
    score += 0.15
elif 40 <= length < 80 or 700 < length <= 1000:
    score += 0.07
```

这能防止回答过短，但也有副作用：模型可能发现“写到 80 字以上就有分”，于是补充重复句、空泛提醒、套话。

比如下面回答可能长度达标，但质量并不好：

```text
需要检查显存，需要检查日志，需要检查服务，需要注意风险，
这些都很重要，非常重要，训练前一定要注意，避免问题。
```

所以长度奖励只能作为弱辅助，不能作为主要质量指标。更好的长度规则应该看“信息密度”，例如：

| 问题 | 改进方向 |
|---|---|
| 只是凑字数 | 加重复惩罚 |
| 长但没有新信息 | 加 unique concept 覆盖 |
| 短但概念准确 | 不应该被过度扣分 |
| 操作题过短 | 可以扣分，因为漏步骤风险更高 |

对你当前项目来说，长度奖励应该比“事实正确、风险边界、不重复”权重更低。

### 7.5 如果模型开始堆关键词，下一轮 reward 怎么改？

一句话回答：

> 如果模型堆关键词，要降低关键词覆盖权重，增加语义正确性、反重复、解释质量和负样本惩罚。

堆关键词的典型输出是：

```text
nvidia-smi 显存 CUDA_VISIBLE_DEVICES 日志 风险 检查 验证 记录。
```

这种回答在当前 v1 reward 下可能拿到不低分，因为它命中了 `required` 和安全词，但它不是好答案。

下一轮 reward 可以这样改：

| 修改点 | 目的 |
|---|---|
| 降低 `required` 总权重 | 避免关键词覆盖支配总分 |
| 给关键词上下文加要求 | 关键词必须出现在解释句或操作句中 |
| 增加重复惩罚 | 惩罚连续重复、词语堆叠、低信息密度 |
| 增加最小句子数但不强模板 | 让回答必须有完整解释 |
| 增加错误概念惩罚 | 例如把 `top_p` 解释成“前 p 个 token”直接扣分 |
| 加入 DPO 失败样本 | 把堆关键词回答作为 rejected |

一个更严格的方向是：不要只检查 `required` 是否出现，而是检查“概念是否被正确使用”。

例如 `nvidia-smi` 不是出现就加满分，而是要和“查看显存、进程、已有服务占用”同时出现，才认为它表达了正确操作。

### 7.6 如果模型回答正确但没出现指定词，应该改数据还是改 reward？

一句话回答：

> 先判断缺失的是“必须术语”还是“同义表达”。如果是同义表达，优先改 reward；如果是任务必须术语，改数据或 prompt。

举例：

| 情况 | 应该怎么改 |
|---|---|
| 回答说“查看 GPU 占用”，但没说 `nvidia-smi` | 改 reward，加入同义词或等价表达 |
| 回答说“限制训练卡”，但没说 `CUDA_VISIBLE_DEVICES` | 看任务要求；如果是命令教学，应改数据要求它写出命令 |
| 回答正确解释 `top_p`，但没出现 `nucleus sampling` | 改 reward，不应该强制英文术语 |
| 回答绕开了关键工具名，导致不可操作 | 改数据和 prompt，明确要求给出工具或命令 |

判断原则：

1. 如果用户真实场景需要这个词才能操作，比如 `CUDA_VISIBLE_DEVICES`，它可以保留在 `required`。
2. 如果只是表达方式不同，比如“核心采样”和 `nucleus sampling`，应该让 reward 支持同义词。
3. 如果模型经常答对但拿低分，说明 reward 太僵硬。
4. 如果模型经常不写关键命令，说明数据或 prompt 没把任务要求说清楚。

所以不是固定“改数据”或“改 reward”。更严格的流程是：

1. 把低分但人工认为正确的回答收集成 failure cases。
2. 标注它为什么正确、缺了哪个指定词。
3. 如果是同义表达，扩展 reward。
4. 如果是任务必需项，扩展训练数据和 prompt。
5. 重新跑 reward selftest，确认好答案得分高、坏答案得分低。

## 8. 下一轮 reward 改进方向

第一轮只做规则 reward。后续可以逐步升级：

| 版本 | 改法 |
|---|---|
| v1 | 当前关键词 + 格式 + 安全规则 |
| v2 | 增加语义同义词，例如 `GPU 占用` 等价于 `nvidia-smi` 场景 |
| v3 | 加入人工标注分数，训练 reward model |
| v4 | 用强模型做 judge，但要防止 judge 偏见和不稳定 |
| v5 | 针对真实任务引入可执行验证，例如命令是否安全、JSON 是否合法 |

你现在不要急着上 v4。先把 v1 的失败原因写清楚。
