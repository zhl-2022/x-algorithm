# MovieLens 推荐实验指标解析

这个文档用于给推荐系统初学者解释当前 MovieLens 阶段所有实验：数据报告、指标含义、
模型原理、训练方式、评估方式和最终结论。

如果你是第一次看，建议按下面顺序阅读：

1. 先看“实验总览”，理解我们到底跑了哪些实验。
2. 再看“核心指标速查”，理解 `Recall@20`、`NDCG@20`、`Coverage@20`、`AUC`、`LogLoss` 分别代表什么。
3. 然后按模型顺序看：Popularity、ItemCF、MF、Two-Tower、DNN Ranker、两阶段重排。
4. 最后看 Candidate K 消融，理解为什么候选集不是越大越好。

## 实验总览

MovieLens 阶段的目标不是追求复杂模型，而是跑通一个完整推荐系统闭环：

```text
数据处理
  ↓
非个性化 baseline
  ↓
协同过滤召回
  ↓
可训练 embedding 召回
  ↓
双塔向量召回
  ↓
DNN 排序
  ↓
Two-Tower 召回 + DNN Ranker 重排
  ↓
候选集大小消融实验
```

当前已经完成的实验如下：

| 阶段 | 实验 | 作用 |
|---|---|---|
| 数据 | MovieLens 1M 数据处理 | 将评分数据转换成正反馈推荐任务 |
| Baseline | Popularity | 验证全局热门推荐的最低基准 |
| 召回 | ItemCF | 引入基于物品相似度的个性化召回 |
| 召回 | MF | 引入可训练用户/电影 embedding |
| 召回 | Two-Tower | 引入更接近企业系统的双塔向量召回 |
| 排序 | DNN Ranker | 引入排序模型和 `AUC`、`LogLoss` |
| Pipeline | Two-Tower + DNN Ranker | 跑通召回候选 + 排序重排 |
| 消融 | Candidate K | 分析候选集大小对最终效果和耗时的影响 |

## 核心指标速查

| 指标 | 主要回答的问题 | 越大越好 |
|---|---|---|
| `Recall@20` | 用户真实喜欢的电影有没有被 Top20 找回来 | 是 |
| `HitRate@20` | Top20 里是否至少命中一个真实喜欢电影 | 是 |
| `Precision@20` | 推荐出去的 20 个电影里有多少是真的命中 | 是 |
| `NDCG@20` | 命中的电影是否排得足够靠前 | 是 |
| `Coverage@20` | 推荐结果覆盖了多少电影库，是否只推热门 | 通常越大越好，但要结合准确率 |
| `Candidate Recall@K` | 召回候选集是否包含真实喜欢电影 | 是 |
| `AUC` | 排序模型是否能把正样本排在负样本前面 | 是 |
| `LogLoss` | 排序模型的概率预测误差 | 否，越小越好 |

## 当前阶段结论

MovieLens 阶段已经完成初步验证闭环。这个数据集适合作为推荐系统入门和工程流程验证，
但它只有用户 ID、电影 ID、评分和电影类别，内容理解能力有限。

后续应该迁移到 MIND-small 新闻推荐数据集，在新闻标题、摘要、类别和用户点击历史上做：

```text
内容感知 Two-Tower
内容感知 DNN Ranker
Two-Tower + Ranker 两阶段新闻推荐 pipeline
```

这会更接近 X/Twitter 类信息流推荐系统。

## Popularity Baseline 报告解析

这个报告评估的是一个最简单的推荐 baseline：**Popularity 热门电影推荐**。

它的逻辑是：

1. 只看训练集里的正反馈。
2. 把 `rating >= 4` 当作用户喜欢这部电影。
3. 统计每部电影在训练集中被喜欢的次数。
4. 对每个测试用户，推荐训练集中最热门的 Top 20 部电影。
5. 推荐前会过滤掉该用户训练集中已经喜欢过的电影。
6. 看推荐列表里有没有命中该用户测试集里真实喜欢的电影。

对应代码在 [popularity_baseline.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/popularity_baseline.py)。

**指标含义**

| 指标         |     当前值 | 含义                                          |
| ------------ | ---------: | --------------------------------------------- |
| 模型         | Popularity | 使用“全局热门电影”作为推荐结果，不做个性化  |
| K            |         20 | 每个用户推荐 20 部电影                        |
| 评估用户数   |      6,037 | 测试集中参与评估的用户数                      |
| Recall@20    |   0.067252 | Top 20 推荐中命中用户真实喜欢电影的比例       |
| HitRate@20   |   0.067252 | 有多少用户的 Top 20 至少命中 1 个真实喜欢电影 |
| Precision@20 |   0.003363 | Top 20 推荐里有多少比例是真正相关电影         |
| NDCG@20      |   0.026149 | 命中电影是否排得靠前，越靠前分数越高          |
| Coverage@20  |   0.045326 | 所有推荐结果覆盖了多少比例的电影库            |

**Recall@20**

Recall@20 关注的是“用户真实喜欢的电影，有多少被推荐系统找回来了”。

公式是：

$$
Recall@K = \frac{TopK 推荐中命中的相关物品数}{用户真实相关物品数}
$$

在你当前的 leave-last 切分方式下，每个测试用户通常只有 1 个测试正样本。所以对单个用户来说：

- 如果 Top 20 里命中了这个测试电影，`Recall@20 = 1`
- 如果没命中，`Recall@20 = 0`

最终 `0.067252` 表示：大约 **6.73% 的测试用户，其真实喜欢的下一部电影出现在热门 Top 20 推荐里**。

这个值不高是正常的，因为 Popularity 是非个性化 baseline，它给所有用户推荐的基本都是热门电影。

**HitRate@20**

HitRate@20 关注的是“Top 20 里有没有命中至少一个相关物品”。

公式是：

$$
HitRate@K =
\begin{cases}
1, & TopK 中至少命中 1 个相关物品 \\
0, & TopK 中没有命中相关物品
\end{cases}
$$

然后对所有用户取平均。

你这里 `HitRate@20 = 0.067252`，和 `Recall@20` 完全相同，原因是：当前测试集中每个用户基本只有 1 个相关电影。

如果一个用户只有 1 个测试正样本，那么：

- 命中就是 `Recall = 1` 且 `HitRate = 1`
- 没命中就是 `Recall = 0` 且 `HitRate = 0`

所以二者相等。

后续如果一个用户测试集中有多个相关物品，Recall 和 HitRate 就会不同。

**Precision@20**

Precision@20 关注的是“推荐出去的 20 个电影里面，有多少是真正命中的”。

公式是：

$$
Precision@K = \frac{TopK 推荐中命中的相关物品数}{K}
$$

当前 `K = 20`。

因为每个用户通常只有 1 个测试正样本，所以单个用户最多命中 1 个电影：

- 命中 1 个：`Precision@20 = 1 / 20 = 0.05`
- 没命中：`Precision@20 = 0`

整体 `Precision@20 = 0.003363`。

它大约等于：

$$
\frac{HitRate@20}{20} = \frac{0.067252}{20} = 0.0033626
$$

也就是说，平均来看，推荐出去的 20 部电影中，真正命中的比例只有 **0.336%**。

注意：在推荐召回任务里，Precision@K 经常会比较低，尤其是测试集每个用户只有一个正样本时。这个指标不是说推荐结果完全没价值，而是说明“20 个位置里只有极少数能被离线测试集确认命中”。

**NDCG@20**

NDCG@20 衡量的不只是“有没有命中”，还看“命中的电影排在第几位”。

NDCG 的核心思想是：

- 命中排第 1 位，价值最高
- 命中排第 2 位，价值低一点
- 命中排第 20 位，价值更低
- 没命中，价值为 0

代码里的 DCG 计算是：

$$
DCG@K = \sum_{i=1}^{K} \frac{rel_i}{\log_2(i + 1)}
$$

其中：

- $i$ 是推荐位置，从 1 开始
- $rel_i$ 表示第 $i$ 个推荐是否命中，命中为 1，否则为 0

然后再除以理想排序下的 IDCG，得到 NDCG：

$$
NDCG@K = \frac{DCG@K}{IDCG@K}
$$

当前 `NDCG@20 = 0.026149`，说明不仅命中率不高，而且命中的电影很多并没有排在非常靠前的位置。

举例：

- 如果用户真实喜欢的电影排在第 1 位，单用户 `NDCG@20 = 1`
- 如果排在第 2 位，分数约为 `1 / log2(3) = 0.6309`
- 如果排在第 10 位，分数约为 `1 / log2(11) = 0.2891`
- 如果没进 Top 20，分数为 `0`

所以 NDCG 比 Recall 更严格，它更接近真实推荐体验：用户通常只会看靠前的推荐。

**Coverage@20**

Coverage@20 衡量推荐结果覆盖了多少物品。

代码里的计算方式是：

$$
Coverage@K = \frac{所有用户 TopK 推荐中出现过的不同电影数}{电影总数}
$$

当前：

```text
Coverage@20 = 0.045326
```

MovieLens 1M 里元数据电影数是 `3,883`，所以推荐覆盖的不同电影数大约是：

$$
3883 \times 0.045326 \approx 176
$$

也就是说，虽然给 6,037 个用户都推荐 Top 20，但由于 Popularity baseline 总是推荐热门电影，最终所有推荐结果加起来只覆盖了大约 **176 部不同电影**。

这体现了热门推荐的典型问题：

- 优点：简单、稳定、适合冷启动 baseline
- 缺点：个性化弱，长尾覆盖差，容易只推少数热门电影

**训练集中最热门的电影**

下面这张表不是评测指标，而是解释 Popularity baseline 的推荐来源。

| 列             | 含义                                      |
| -------------- | ----------------------------------------- |
| 排名           | 按训练集正反馈次数从高到低排序            |
| 电影 ID        | MovieLens 中的电影编号                    |
| 训练集正反馈数 | 有多少训练集用户给这部电影打了 `>=4` 分 |
| 标题           | 电影名称                                  |
| 类别           | 电影类型标签                              |

例如：

```text
American Beauty (1999) 训练集正反馈数 2,800
```

表示在训练集中，有 2,800 条正反馈交互喜欢这部电影，所以它是当前最热门电影。

**一句话总结**

这个报告说明：只靠全局热门电影推荐，Top 20 能命中约 **6.73%** 用户的下一部喜欢电影，覆盖约 **4.53%** 的电影库。它是一个合格的 baseline，但个性化能力很弱，后续 ItemCF、MF、Two-Tower 的目标就是在 `Recall@20`、`NDCG@20` 上超过它。



## MovieLens 1M 数据报告

这个 `data_report.md` 是“数据体检报告”：它不是模型效果报告，而是在回答 **MovieLens 1M 这个数据集有多大、交互有多稀疏、正反馈如何定义、训练/验证/测试怎么切分**。

生成逻辑在 [prepare_movielens.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/prepare_movielens.py)。

**1. 数据集概览**

| 指标               |    当前值 | 含义                                       |
| ------------------ | --------: | ------------------------------------------ |
| 原始评分数         | 1,000,209 | `ratings.dat` 中总共有多少条用户评分记录 |
| 有评分行为的用户数 |     6,040 | 至少给过一部电影评分的用户数量             |
| 元数据中的电影数   |     3,883 | `movies.dat` 中记录的电影总数            |
| 有评分的电影数     |     3,706 | 至少被一个用户评分过的电影数量             |
| 评分矩阵密度       |   4.2647% | 用户-电影评分矩阵里，实际有评分的位置占比  |
| 评分矩阵稀疏度     |  95.7353% | 用户-电影评分矩阵里，没有评分的位置占比    |

推荐系统里常把数据想象成一个矩阵：

```text
          movie_1  movie_2  movie_3 ...
user_1       5       空       4
user_2       空      3       空
user_3       4       空       空
```

如果有 `6,040` 个用户、`3,883` 部电影，理论上最多有：

```text
6,040 * 3,883 = 23,453,320
```

个用户-电影评分位置。

实际只有 `1,000,209` 条评分，所以：

```text
评分矩阵密度 = 1,000,209 / (6,040 * 3,883)
           ≈ 0.042647
           = 4.2647%
```

```text
评分矩阵稀疏度 = 1 - 4.2647%
              = 95.7353%
```

这说明绝大多数用户并没有看过或评分过绝大多数电影。推荐系统的数据天然就是稀疏的。

**2. 元数据中的电影数 vs 有评分的电影数**

`元数据中的电影数 = 3,883`，来自 `movies.dat`。

`有评分的电影数 = 3,706`，来自 `ratings.dat` 中出现过的 `item_id` 去重数量。

两者不同，说明有：

```text
3,883 - 3,706 = 177
```

部电影在元数据里存在，但没有任何评分记录。

后续评估 `Coverage@K` 时，用的是完整电影元数据数量 `3,883` 作为物品全集。

**3. 正反馈定义**

报告里写：

```text
本实验将 rating >= 4 定义为正反馈。
```

MovieLens 是显式评分数据，评分范围是 `1` 到 `5`。但推荐召回任务通常更关心“用户是否喜欢/是否会点击/是否会互动”，所以这里把评分转换成隐式反馈：

```text
rating >= 4  → 用户喜欢这部电影，记为正反馈
rating < 4   → 不作为正反馈
```

注意：当前第一阶段实验没有把 `rating < 4` 当作负样本，只是先用正反馈做召回评估。

**4. 正反馈交互数**

| 指标             |   当前值 | 含义                                  |
| ---------------- | -------: | ------------------------------------- |
| 正反馈交互数     |  575,281 | 所有 `rating >= 4` 的评分记录数     |
| 有正反馈的用户数 |    6,038 | 至少有一条 `rating >= 4` 的用户数   |
| 有正反馈的电影数 |    3,533 | 至少被一个用户打过 `>=4` 分的电影数 |
| 正反馈矩阵密度   |  2.4537% | 正反馈在用户-电影矩阵中的占比         |
| 正反馈矩阵稀疏度 | 97.5463% | 用户-电影矩阵里没有正反馈的位置占比   |

原始有 `6,040` 个评分用户，但有正反馈的用户只有 `6,038` 个，说明有 2 个用户虽然打过分，但没有任何电影评分达到 `4` 或 `5`。

正反馈矩阵密度的计算方式是：

```text
正反馈矩阵密度 = 正反馈交互数 / (有正反馈用户数 * 元数据电影数)
```

代入当前数据：

```text
575,281 / (6,038 * 3,883)
≈ 0.024537
= 2.4537%
```

正反馈矩阵稀疏度：

```text
1 - 2.4537% = 97.5463%
```

这里用 `3,883` 作为电影总数，而不是 `3,533`。也就是说，它把所有电影元数据都看作推荐候选空间，这更贴近后续推荐系统评估。

**5. 交互分布**

| 指标                   | 当前值 | 计算方式                      |
| ---------------------- | -----: | ----------------------------- |
| 用户平均评分数         | 165.60 | 原始评分数 / 有评分用户数     |
| 有评分电影的平均评分数 | 269.89 | 原始评分数 / 有评分电影数     |
| 正反馈用户平均正反馈数 |  95.28 | 正反馈交互数 / 有正反馈用户数 |
| 正反馈电影平均正反馈数 | 162.83 | 正反馈交互数 / 有正反馈电影数 |

具体计算：

```text
用户平均评分数 = 1,000,209 / 6,040 ≈ 165.60
```

表示每个用户平均打过约 166 次分。

```text
有评分电影的平均评分数 = 1,000,209 / 3,706 ≈ 269.89
```

表示每部有评分电影平均收到约 270 条评分。

```text
正反馈用户平均正反馈数 = 575,281 / 6,038 ≈ 95.28
```

表示每个有正反馈的用户平均喜欢约 95 部电影。

**6. 评分分布**

| 评分 |    数量 |     占比 |
| ---: | ------: | -------: |
|    1 |  56,174 |  5.6162% |
|    2 | 107,557 | 10.7535% |
|    3 | 261,197 | 26.1142% |
|    4 | 348,971 | 34.8898% |
|    5 | 226,310 | 22.6263% |

每一行的占比计算方式是：

```text
某评分占比 = 该评分数量 / 原始评分总数
```

比如评分 `4`：

```text
348,971 / 1,000,209 ≈ 34.8898%
```

正反馈由评分 `4` 和 `5` 组成：

```text
348,971 + 226,310 = 575,281
```

这正好对应报告里的：

```text
正反馈交互数 = 575,281
```

从分布看，MovieLens 1M 里的评分整体偏正向：

```text
4 分 + 5 分 = 34.8898% + 22.6263% = 57.5161%
```

也就是说，超过一半评分被当前实验视为正反馈。

**7. 按时间顺序切分**

| 数据集 |  交互数 | 用户数 |
| ------ | ------: | -----: |
| 训练集 | 563,209 |  6,038 |
| 验证集 |   6,035 |  6,035 |
| 测试集 |   6,037 |  6,037 |

这里的切分只使用正反馈，不使用所有评分。

脚本按每个用户的正反馈时间序列排序：

```python
key=lambda row: (row.timestamp, row.item_id)
```

然后按规则切分：

```text
如果用户正反馈数 >= 3：
  训练集 = 除最后两条之外的所有正反馈
  验证集 = 倒数第 2 条正反馈
  测试集 = 最后 1 条正反馈

如果用户正反馈数 == 2：
  训练集 = 第 1 条正反馈
  测试集 = 第 2 条正反馈
  验证集 = 无

如果用户正反馈数 == 1：
  训练集 = 这 1 条正反馈
  验证集 = 无
  测试集 = 无
```

所以：

```text
训练集用户数 = 6,038
```

因为所有有正反馈的用户至少会有 1 条进入训练集。

```text
验证集用户数 = 6,035
```

说明有 `6,035` 个用户至少有 3 条正反馈。

```text
测试集用户数 = 6,037
```

说明有 `6,037` 个用户至少有 2 条正反馈。

结合正反馈用户总数 `6,038` 可以反推出：

```text
有 6,035 个用户 >= 3 条正反馈
有 2 个用户正好 2 条正反馈
有 1 个用户正好 1 条正反馈
```

因为：

```text
6,035 + 2 + 1 = 6,038
```

**8. 为什么要按时间顺序切分**

推荐系统不能用“未来行为”预测“过去行为”。

比如一个用户的正反馈序列是：

```text
A -> B -> C -> D -> E
```

当前切分方式是：

```text
训练集：A, B, C
验证集：D
测试集：E
```

这样模型学习的是用户过去喜欢过什么，然后预测用户未来可能喜欢什么。

这比随机切分更合理。随机切分可能把用户后来的行为放进训练集，再去预测用户早期行为，会造成时间穿越，离真实线上推荐更远。

**9. 这个报告对后续实验有什么用**

这个报告提供了后续所有模型实验的基准上下文：

- 数据是否足够大：`1,000,209` 条评分，足够做入门推荐实验。
- 正反馈是否足够多：`575,281` 条正反馈，足够训练召回模型。
- 数据是否稀疏：正反馈矩阵稀疏度 `97.5463%`，说明推荐任务确实有难度。
- 评估用户有多少：测试集 `6,037` 个用户，后续 `Recall@20` 有稳定评估基础。
- 是否存在长尾问题：电影平均正反馈数和 Coverage 后续可以一起分析热门偏置。

一句话总结：`data_report.md` 说明 MovieLens 1M 被转换成了一个正反馈推荐任务，包含 `6,038` 个正反馈用户、`575,281` 条正反馈交互，并按时间顺序切成 train/valid/test，为后续 Popularity、ItemCF、MF、Two-Tower 的离线评测提供统一数据基础。



## ItemCF Baseline 报告解析

这个报告评估的是第二个推荐 baseline：**ItemCF 基于物品的协同过滤推荐**。

Popularity 只知道“全站最热门的电影是什么”，不关心用户是谁。ItemCF 则开始做个性化：

```text
如果一个用户喜欢过 Toy Story、Matrix、Star Wars，
那就给他推荐和这些电影相似的其他电影。
```

对应代码在 [itemcf_baseline.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/itemcf_baseline.py)。

**这次 ItemCF 实验的核心流程**

```text
训练集正反馈
  ↓
统计每部电影被多少用户喜欢
  ↓
统计每两部电影被同一批用户同时喜欢的次数
  ↓
计算电影-电影相似度
  ↓
根据用户历史喜欢电影召回相似电影
  ↓
过滤用户已经看过的电影
  ↓
取 Top 20 作为推荐结果
  ↓
用测试集最后一次正反馈评估命中情况
```

当前 ItemCF 结果是：

|---|---:|---|
| 模型 | ItemCF | 基于物品相似度的个性化推荐 |

| HitRate@20 | 0.082160 | 有多少用户的 Top 20 至少命中 1 个真实喜欢电影 |
| Precision@20 | 0.004108 | Top 20 推荐里有多少比例是真正相关电影 |
| NDCG@20 | 0.032793 | 命中电影是否排得靠前 |

和 Popularity 相比：

|---|---:|---:|---:|
| Popularity | 0.067252 | 0.026149 | 0.045326 |
| ItemCF | 0.082160 | 0.032793 | 0.132372 |

这说明 ItemCF 相比热门推荐：

- 更容易命中用户未来喜欢的电影。
- 命中的电影平均排位更靠前。
- 推荐覆盖的电影更多，不再只集中在少数热门电影上。

**1. ItemCF 里的“评分”到底是什么**

这里容易混淆，因为 ItemCF 里其实有两种分数：

1. 电影和电影之间的相似度分数。
2. 某个用户对候选电影的推荐得分。

公式：

```text
co_count(i, j) / sqrt(count(i) * count(j))
```

计算的是 **电影 i 和电影 j 之间的相似度分数**，不是用户对电影的最终推荐得分。

然后用户的最终推荐得分是：

```text
score(user, candidate) = 用户历史电影与 candidate 的相似度累加
```

所以 ItemCF 分两步：

```text
先算电影之间像不像
再根据用户历史喜欢过什么，给候选电影打推荐分
```

**2. 每个电影都进行计算吗**

准确说，不是“每个电影和所有电影强行两两计算”，而是：

1. 先遍历每个用户在训练集中喜欢过的电影集合。
2. 对同一个用户喜欢过的电影做两两配对。
3. 只要两部电影被同一个用户同时喜欢过，它们就产生一次共现。
4. 最后只对有共现关系的电影对计算相似度。

举个极简例子：

```text
用户 A 喜欢：电影 1、电影 2、电影 3
用户 B 喜欢：电影 1、电影 2
用户 C 喜欢：电影 2、电影 4
```

那么共现统计是：

```text
电影 1 和电影 2：共同出现 2 次，来自用户 A、B
电影 1 和电影 3：共同出现 1 次，来自用户 A
电影 2 和电影 3：共同出现 1 次，来自用户 A
电影 2 和电影 4：共同出现 1 次，来自用户 C
```

如果两部电影从来没有被任何同一个用户同时喜欢过，它们就没有共现记录，也不会进入相似度候选。

代码里对应逻辑是：

```python
for user_items in train_by_user.values():
    sorted_items = sorted(user_items)
    item_counts.update(sorted_items)
    for index, item_i in enumerate(sorted_items):
        for item_j in sorted_items[index + 1:]:
            co_counts[item_i][item_j] += 1
            co_counts[item_j][item_i] += 1
```

**3. `count(i)` 是什么**

`count(i)` 表示喜欢电影 $i$ 的训练用户数。

例如：

```text
count(Toy Story) = 多少训练用户给 Toy Story 打过 >=4 分
count(Matrix) = 多少训练用户给 Matrix 打过 >=4 分
```

它不是电影的总评分数，而是在当前实验里“正反馈训练集中的被喜欢次数”。

**4. `co_count(i, j)` 是什么**

`co_count(i, j)` 表示同时喜欢电影 $i$ 和电影 $j$ 的训练用户数。

例如：

```text
co_count(Toy Story, Toy Story 2)
= 多少训练用户既喜欢 Toy Story，也喜欢 Toy Story 2
```

这个值越大，说明两部电影越经常被同一批用户一起喜欢。

**5. 为什么不能只用 `co_count(i, j)`**

如果只用共现次数，热门电影会天然占便宜。

比如《Star Wars》很热门，它可能和很多电影都有很高共现次数，因为喜欢它的人太多了。
这会导致 ItemCF 变成另一种“热门推荐”，个性化会变弱。

所以要除以：

```text
sqrt(count(i) * count(j))
```

这个分母表示两部电影各自的热门程度。除以它之后，可以降低热门电影的天然优势。

完整公式是：

$$
sim(i, j) = \frac{co\_count(i, j)}{\sqrt{count(i) \times count(j)}}
$$

这其实就是“电影-用户二值向量”的余弦相似度。

可以这样理解：

```text
电影 i = 哪些用户喜欢过它
电影 j = 哪些用户喜欢过它
sim(i, j) = 这两个用户集合有多像
```

**6. 一个具体小例子**

假设：

```text
count(电影 1) = 100
count(电影 2) = 25
co_count(电影 1, 电影 2) = 10
```

那么：

$$
sim(1, 2) = \frac{10}{\sqrt{100 \times 25}}
          = \frac{10}{50}
          = 0.2
$$

这个 `0.2` 表示电影 1 和电影 2 的相似度。

再假设：

```text
count(电影 3) = 10,000
count(电影 2) = 25
co_count(电影 3, 电影 2) = 50
```

虽然电影 3 和电影 2 的共现次数更高：

```text
50 > 10
```

但相似度是：

$$
sim(3, 2) = \frac{50}{\sqrt{10000 \times 25}}
          = \frac{50}{500}
          = 0.1
$$

也就是说，电影 3 可能只是因为太热门才和电影 2 一起出现很多次，并不一定更相似。

**7. 用户的推荐分是怎么来的**

电影相似度算完以后，开始给用户推荐。

假设某个用户训练集中喜欢过：

```text
Toy Story
Matrix
Star Wars
```

系统会分别找：

```text
和 Toy Story 相似的电影
和 Matrix 相似的电影
和 Star Wars 相似的电影
```

如果某个候选电影同时和多个历史电影相似，它的分数会累加。

例如：

```text
score(用户 U, 候选电影 X)
= sim(Toy Story, X)
+ sim(Matrix, X)
+ sim(Star Wars, X)
```

如果：

```text
sim(Toy Story, X) = 0.30
sim(Matrix, X) = 0.10
sim(Star Wars, X) = 0.20
```

那么：

```text
score(U, X) = 0.30 + 0.10 + 0.20 = 0.60
```

这个 `0.60` 才是“用户 U 对候选电影 X 的推荐得分”。

代码里对应逻辑是：

```python
for history_item in history_items:
    for candidate_item, similarity in similarities.get(history_item, []):
        if candidate_item in history_items:
            continue
        candidate_scores[candidate_item] += similarity
```

其中：

- `history_item` 是用户训练集中喜欢过的电影。
- `candidate_item` 是和历史电影相似的候选电影。
- `candidate_scores[candidate_item] += similarity` 表示把相似度累加到候选电影得分上。

**8. 为什么要过滤用户已经看过的电影**

如果用户训练集中已经喜欢过 Toy Story，就没必要在测试时再把 Toy Story 推荐给他。

所以代码里有：

```python
if candidate_item in history_items:
    continue
```

这表示：

```text
候选电影如果已经在用户历史里出现过，就跳过，不推荐。
```

这样评估更接近真实推荐场景：我们希望推荐用户未来可能喜欢、但还没出现在训练历史里的电影。

**9. 每个物品保留相似物品数是什么意思**

报告里有：

```text
每个物品保留相似物品数 = 100
```

这是为了控制计算量。

对每部电影，可能有很多共现电影。我们不会保存所有相似电影，而是只保留相似度最高的 100 部：

```python
similarities[item_i] = scored_neighbors[:top_similar_per_item]
```

这类似召回系统中的候选裁剪：

```text
每个历史电影只贡献最相关的一批候选电影
```

**12. 一句话总结**

ItemCF 的核心不是“给电影训练一个复杂模型”，而是：


真正给用户推荐时，还要把用户历史电影与候选电影的相似度累加，得到用户对候选电影的最终推荐得分。



## Matrix Factorization Baseline 报告解析

这个报告评估的是第三个推荐模型：**Matrix Factorization，矩阵分解，简称 MF**。

Popularity 和 ItemCF 都不需要训练神经网络：

- Popularity 只统计全局热门电影。
- ItemCF 只统计电影之间的共现相似度。

MF 开始进入“可训练模型”阶段。它会给每个用户学一个向量，也给每部电影学一个向量，然后用这两个向量的相似程度预测用户是否可能喜欢这部电影。

对应代码在 [mf_train.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/mf_train.py)。

**这次 MF 实验的核心流程**

```text
训练集正反馈
  ↓
为每个用户建立 user embedding
  ↓
为每部电影建立 item embedding
  ↓
构造正样本和随机负样本
  ↓
用 BCE Loss 训练用户向量和电影向量
  ↓
对每个测试用户给全量电影打分
  ↓
过滤训练集中已看过电影
  ↓
取 Top 20 作为推荐结果
  ↓
用测试集最后一次正反馈评估命中情况
```

当前 MF 结果是：

| 指标 | 当前值 | 含义 |
|---|---:|---|
| 模型 | MF | 用用户向量和电影向量点积做推荐 |
| K | 20 | 每个用户推荐 20 部电影 |
| 评估用户数 | 6,037 | 测试集中参与评估的用户数 |
| Recall@20 | 0.104025 | Top 20 推荐命中用户真实喜欢电影的比例 |
| HitRate@20 | 0.104025 | 有多少用户的 Top 20 至少命中 1 个真实喜欢电影 |
| Precision@20 | 0.005201 | Top 20 推荐里有多少比例是真正相关电影 |
| NDCG@20 | 0.039807 | 命中电影是否排得靠前 |
| Coverage@20 | 0.481844 | 所有推荐结果覆盖了多少比例的电影库 |

和前两个 baseline 对比：

| 模型 | Recall@20 | NDCG@20 | Coverage@20 |
|---|---:|---:|---:|
| Popularity | 0.067252 | 0.026149 | 0.045326 |
| ItemCF | 0.082160 | 0.032793 | 0.132372 |
| MF | 0.104025 | 0.039807 | 0.481844 |

这个结果说明：

- MF 的 `Recall@20` 高于 Popularity 和 ItemCF，说明它更容易找回用户未来喜欢的电影。
- MF 的 `NDCG@20` 高于 Popularity 和 ItemCF，说明命中的电影整体排得更靠前。
- MF 的 `Coverage@20` 明显更高，说明它不再只集中推荐少数热门电影。

**1. MF 到底在学习什么**

MF 的核心假设是：

```text
用户的兴趣可以用一个低维向量表示。
电影的内容/受众偏好也可以用一个低维向量表示。
如果用户向量和电影向量方向接近，用户就更可能喜欢这部电影。
```

在当前实验里：

```text
Embedding Dim = 64
```

表示每个用户会被表示成一个 64 维向量，每部电影也会被表示成一个 64 维向量。

可以简单理解为：

```text
用户 U = [0.12, -0.03, 0.44, ...]
电影 I = [0.10, -0.08, 0.39, ...]
```

这些数字不是人工指定的，而是在训练过程中自动学出来的。

**2. MF 的推荐分是怎么计算的**

MF 对用户 $u$ 和电影 $i$ 的打分公式可以理解为：

$$
score(u, i) = p_u^\top q_i + b_u + b_i + b_0
$$

其中：

- $p_u$ 是用户 $u$ 的 embedding。
- $q_i$ 是电影 $i$ 的 embedding。
- $p_u^\top q_i$ 是用户向量和电影向量的点积。
- $b_u$ 是用户偏置，表示某些用户整体更容易给出正反馈。
- $b_i$ 是电影偏置，表示某些电影整体更容易被喜欢。
- $b_0$ 是全局偏置。

代码里对应的核心逻辑是：

```python
user_vectors = self.user_embedding(user_indices)
item_vectors = self.item_embedding(item_indices)
dot = (user_vectors * item_vectors).sum(dim=1)
return dot + user_bias + item_bias + self.global_bias
```

注意：这个 `score` 不是 MovieLens 原始评分的 1 到 5 分，而是模型内部用于排序的预测分。分数越高，模型越认为用户可能喜欢这部电影。

**3. 正样本是怎么来的**

正样本来自训练集里的正反馈：

```text
rating >= 4
```

例如：

```text
用户 10 给电影 589 打了 5 分
```

在当前实验里会被转换成：

```text
(user_id=10, item_id=589, label=1)
```

也就是说，模型要学习：

```text
用户 10 应该喜欢电影 589
```

**4. 负样本是怎么来的**

MovieLens 原始数据里没有明确告诉我们“用户一定不喜欢哪部电影”。所以当前实验使用随机负采样。

规则是：

```text
对每个正样本，从该用户没有出现过正反馈的电影里随机采一部，作为负样本。
```

例如用户 10 的正反馈历史是：

```text
电影 A、电影 B、电影 C
```

那么负样本会从：

```text
不在 A、B、C 里的其他电影
```

随机采样。

在当前配置中：

```text
每个正样本负采样数 = 1
```

所以训练数据大致变成：

```text
1 条正样本 + 1 条负样本
```

代码里对应的是：

```python
if candidate not in user_positive_items:
    return candidate
```

需要注意：随机负样本不代表用户真的讨厌这部电影，只代表“训练集中没有证据表明用户喜欢它”。这是隐式反馈推荐里很常见的近似做法。

**5. MF 是怎么训练的**

当前 MF 使用的是二分类训练目标。

模型输出一个分数 $score(u, i)$，然后用 `BCEWithLogitsLoss` 判断这个分数是否能区分正负样本：

```text
正样本 label = 1

```

训练目标是：

```text
让正样本分数变高，让负样本分数变低。
```

| 评估耗时（秒） | 5.45 | 全量电影打分和指标计算耗时 |

**6. MF 评估时是怎么推荐的**

训练完成后，评估流程是：

1. 对每个测试用户取出用户 embedding。
2. 用这个用户 embedding 和所有电影 embedding 做打分。
3. 把用户训练集中已经看过的电影分数设为负无穷，避免重复推荐。
4. 取分数最高的 Top 20 部电影。
5. 看 Top 20 里是否包含测试集那部真实喜欢的电影。

代码里的核心评估逻辑是：

```python
scores = model.score_user(user_to_index[user_id], device=device).detach().cpu()
for seen_item in train_by_user.get(user_id, set()):
    scores[seen_index] = -float("inf")
top_indices = torch.topk(scores, k=min(k, len(scores))).indices.tolist()
```

所以 MF 和 ItemCF 最大的区别是：

```text
ItemCF 的分数来自统计相似度累加。
MF 的分数来自训练出来的用户向量和电影向量。
```

**8. 为什么 MF 比 ItemCF 更强**

ItemCF 只能利用“电影和电影是否经常被同一批用户喜欢”。

MF 会同时学习：

- 用户之间的相似兴趣。
- 电影之间的潜在相似性。
- 用户和电影之间的匹配关系。
- 用户偏置和电影偏置。

所以即使两部电影没有非常明显的直接共现，MF 也可能通过 embedding 空间学到它们和某类用户之间的共同关系。

这就是为什么当前实验里：

```text
MF Recall@20 = 0.104025
ItemCF Recall@20 = 0.082160
```

MF 明显更高。

**9. 一句话总结**

MF 的核心是：

```text
给每个用户和每部电影学习一个向量，
用向量点积预测用户是否喜欢电影，
再按预测分数给每个用户推荐 Top 20。
```

它是当前项目里第一个真正通过梯度下降训练出来的推荐模型，也是在 MLU 容器中跑通训练闭环的第一步。



## Two-Tower 召回模型报告解析

这个报告评估的是第四个推荐模型：**Two-Tower 双塔召回模型**。

Two-Tower 是企业推荐系统里非常常见的召回结构。它的核心思想是：

```text
用户塔负责把用户变成一个向量。
物品塔负责把电影变成一个向量。
推荐时计算用户向量和电影向量的相似度。
```

对应代码在 [two_tower_train.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/two_tower_train.py)。

**这次 Two-Tower 实验的核心流程**

```text
训练集正反馈
  ↓
构建用户 ID embedding
  ↓
构建用户历史电影 embedding 均值
  ↓
用户塔生成 user vector
  ↓
物品塔生成 item vector
  ↓
构造正样本和随机负样本
  ↓
用 BCE Loss 训练双塔向量空间
  ↓
评估时计算用户向量与全量电影向量相似度
  ↓
过滤训练集中已看过电影
  ↓
取 Top 20 作为推荐结果
  ↓
用测试集最后一次正反馈评估命中情况
```

当前 Two-Tower 正式结果是：

| 指标 | 当前值 | 含义 |
|---|---:|---|
| 模型 | Two-Tower | 用户塔和物品塔分别生成向量 |
| K | 20 | 每个用户推荐 20 部电影 |
| 评估用户数 | 6,037 | 测试集中参与评估的用户数 |
| Recall@20 | 0.101872 | Top 20 推荐命中用户真实喜欢电影的比例 |
| HitRate@20 | 0.101872 | 有多少用户的 Top 20 至少命中 1 个真实喜欢电影 |
| Precision@20 | 0.005094 | Top 20 推荐里有多少比例是真正相关电影 |
| NDCG@20 | 0.040244 | 命中电影是否排得靠前 |
| Coverage@20 | 0.340973 | 所有推荐结果覆盖了多少比例的电影库 |

和已有模型对比：

| 模型 | Recall@20 | NDCG@20 | Coverage@20 |
|---|---:|---:|---:|
| Popularity | 0.067252 | 0.026149 | 0.045326 |
| ItemCF | 0.082160 | 0.032793 | 0.132372 |
| MF | 0.104025 | 0.039807 | 0.481844 |
| Two-Tower | 0.101872 | 0.040244 | 0.340973 |

这个结果说明：

- Two-Tower 的 `Recall@20` 接近 MF，略低于 MF。
- Two-Tower 的 `NDCG@20` 略高于 MF，说明命中电影的位置质量更好。
- Two-Tower 的 `Coverage@20` 高于 ItemCF，但低于 MF，说明推荐覆盖面处于中间水平。

**1. Two-Tower 为什么叫“双塔”**

因为它有两个相对独立的网络：

```text
用户塔：user features -> user vector
物品塔：item features -> item vector
```

当前这个双塔 **不是 Transformer 架构**，没有 Transformer Encoder、Self-Attention、Multi-Head Attention，也没有 token 序列建模。

它是一个传统推荐系统里常见的 `Embedding + MLP` 双塔结构：

| 组件 | 当前结构 |
|---|---|
| 用户 ID embedding | 128 维 |
| 用户历史电影 embedding 均值 | 128 维 |
| 用户塔输入维度 | 256 维 |
| 用户塔 MLP | `Linear(256 -> 256)`、`ReLU`、`Dropout(0.0)`、`Linear(256 -> 128)` |
| 电影 ID embedding | 128 维 |
| 物品塔 MLP | `Linear(128 -> 256)`、`ReLU`、`Dropout(0.0)`、`Linear(256 -> 128)` |
| 双塔输出维度 | 128 维 |
| 打分方式 | L2 归一化后点积，再除以 `temperature=0.1` |

如果按线性层来数：

```text
用户塔：2 个 Linear 层
物品塔：2 个 Linear 层
双塔合计：4 个 Linear 层
```

如果按隐藏层来数：

```text
每个塔 1 个隐藏层 + 1 个输出投影层
```

当前实验里的用户塔输入是两部分：

1. `user_id embedding`：表示这个用户本身的 ID 特征。
2. 用户历史正反馈电影 embedding 的均值：表示这个用户过去喜欢过什么电影。

当前实验里的物品塔输入是：

```text
item_id embedding
```

也就是说，用户塔比 MF 多利用了一个明确的历史行为表示：

```text
用户过去喜欢过哪些电影
```

**2. 用户塔输入是怎么构造的**

对每个用户，先拿到训练集里这个用户喜欢过的电影，按时间排序后最多保留最近 100 部：

```text
最大历史电影数 = 100
```

然后把这些历史电影的 embedding 求平均：

$$
h_u = \frac{1}{n}\sum_{i \in history(u)} e_i
$$

其中：

- $h_u$ 是用户历史兴趣向量。
- $history(u)$ 是用户 $u$ 在训练集里的历史正反馈电影集合。
- $e_i$ 是历史电影 $i$ 的 embedding。
- $n$ 是保留下来的历史电影数量。

再把用户 ID embedding 和历史兴趣向量拼接：

$$
x_u = [user\_embedding(u); h_u]
$$

这个拼接后的向量会进入用户塔 MLP，生成最终用户向量。

代码里对应逻辑是：

```python
history_vectors = self.history_embedding(history_items).sum(dim=1) / history_lengths
user_vectors = self.user_embedding(user_indices)
tower_input = torch.cat([user_vectors, history_vectors], dim=1)
return F.normalize(self.user_tower(tower_input), dim=1)
```

**3. 物品塔输入是怎么构造的**

物品塔相对简单。每部电影有一个 `item_id embedding`，再经过 MLP 得到最终电影向量：

```python
item_vectors = self.item_embedding(item_indices)
return F.normalize(self.item_tower(item_vectors), dim=1)
```

当前没有使用电影标题、类别等内容特征。后续迁移到 MIND 新闻推荐时，可以把新闻标题、类别、摘要等内容特征加入物品塔。

**4. Two-Tower 的推荐分是怎么计算的**

用户塔输出用户向量 $u$，物品塔输出电影向量 $v$。两个向量都会做 L2 归一化。

推荐分可以理解为：

$$
score(user, item) = \frac{u^\top v}{temperature}
$$

其中：

- $u^\top v$ 是用户向量和电影向量的点积。
- 因为向量做过归一化，所以点积也可以理解为余弦相似度。
- `temperature` 用来调节分数尺度，当前为 `0.1`。

分数越高，说明用户向量和电影向量越接近，模型越认为用户可能喜欢这部电影。

**5. Two-Tower 是怎么训练的**

当前正式配置使用随机负采样 BCE 训练：

```text
正样本 label = 1
负样本 label = 0
```

正样本来自训练集正反馈：

```text
rating >= 4
```

负样本和 MF 一样，从用户没有出现过正反馈的电影中随机采样。

训练目标是：

```text
让用户向量更接近正样本电影向量，
让用户向量远离负样本电影向量。
```

当前训练配置：

| 配置 | 当前值 | 含义 |
|---|---:|---|
| 训练目标 | bce | 随机负采样二分类训练 |
| 设备 | mlu | 在公司寒武纪 MLU 容器中训练 |
| 用户数 | 6,038 | 用户塔 user embedding 数量 |
| 电影数 | 3,883 | 物品塔 item embedding 数量 |
| Embedding Dim | 128 | 原始用户/电影 embedding 维度 |
| Tower Dim | 128 | 双塔最终输出向量维度 |
| Hidden Dim | 256 | 双塔 MLP 隐藏层维度 |
| 最大历史电影数 | 100 | 用户塔最多使用最近 100 部历史正反馈电影 |
| Epochs | 10 | 完整遍历训练样本 10 轮 |
| Batch Size | 4,096 | 每次送入模型训练的正样本数量 |
| 每个正样本负采样数 | 1 | 每条正样本配 1 条随机负样本 |
| Learning Rate | 0.005 | 优化器学习率 |

**6. 这次实际跑过哪些 Two-Tower 方案**

这次不是只跑了一版，而是先后试了几个方案：

| 方案 | 关键配置 | Recall@20 | NDCG@20 | Coverage@20 | 结论 |
|---|---|---:|---:|---:|---|
| 初始 BCE | `embedding_dim=64`、`negative_samples=3`、`epochs=5` | 0.078681 | 0.031253 | 0.144991 | 跑通流程，但效果弱于 ItemCF |
| In-Batch | `loss=in-batch`、`batch_size=2048`、`epochs=5` | 0.064933 | 0.024501 | 0.975277 | 覆盖率很高，但命中率下降 |
| 调参 BCE | `embedding_dim=64`、`negative_samples=1`、`epochs=8` | 0.093093 | 0.037356 | 0.320113 | 明显优于 ItemCF |
| 正式 BCE | `embedding_dim=128`、`tower_dim=128`、`epochs=10` | 0.101872 | 0.040244 | 0.340973 | 当前采用的正式结果 |

这里有一个重要现象：

```text
Coverage 高不代表推荐效果一定好。
```

`in-batch` 方案的 Coverage 达到 `0.975277`，说明它几乎覆盖了整个电影库，但 `Recall@20` 只有 `0.064933`。这表示它推荐得很分散，但并没有更准确地命中用户未来喜欢的电影。

所以评估推荐系统不能只看单个指标，要同时看：

- `Recall@K`：有没有找回用户真正喜欢的电影。
- `NDCG@K`：命中的电影是否排得靠前。
- `Coverage@K`：推荐结果是否覆盖足够多物品。

**7. Two-Tower 评估时是怎么推荐的**

评估时，Two-Tower 和 MF 一样都会对全量电影打分，但打分方式不同。

MF 是：

```text
用户 embedding 和电影 embedding 点积 + bias
```

Two-Tower 是：

```text
用户塔输出向量 和 物品塔输出向量 做相似度计算
```

代码里的核心评估逻辑是：

```python
item_vectors = model.encode_items(all_item_indices)
user_vector = model.encode_users(user_index)
scores = (user_vector @ item_vectors.T).squeeze(0).detach().cpu()
```

然后同样过滤训练集中已看过的电影：

```python
scores[seen_index] = -float("inf")
```

最后取 Top 20：

```python
top_indices = torch.topk(scores, k=min(k, len(scores))).indices.tolist()
```

**8. Two-Tower 和 MF 有什么区别**

这两个模型看起来都在学用户向量和电影向量，但结构不同。

| 对比点 | MF | Two-Tower |
|---|---|---|
| 用户表示 | 主要是 `user_id embedding` | `user_id embedding` + 历史电影 embedding 均值 |
| 物品表示 | `item_id embedding` | `item_id embedding` 经过物品塔 MLP |
| 打分方式 | 点积 + 用户/电影 bias | 双塔向量点积 / 相似度 |
| 网络结构 | 简单 embedding 模型 | 用户塔和物品塔两个 MLP |
| 工业意义 | 推荐系统基础模型 | 更接近线上向量召回架构 |
| 当前 Recall@20 | 0.104025 | 0.101872 |
| 当前 NDCG@20 | 0.039807 | 0.040244 |

当前 Two-Tower 的 `Recall@20` 没有超过 MF，但它依然很重要，因为它更接近真实企业推荐系统的召回结构。

在线上系统里，物品塔可以提前把所有物品向量算好并写入向量库。请求来时，只需要实时计算用户向量，然后用向量检索快速找 TopK 候选。

也就是：

```text
离线：提前计算 item vectors
在线：实时计算 user vector
检索：user vector 在向量库里找最近的 item vectors
```

这就是 Two-Tower 在工程上的价值。

**10. 当前四个模型应该怎么理解**

可以按“推荐系统能力层级”理解：

| 层级 | 模型 | 学到了什么 |
|---|---|---|
| 非个性化 | Popularity | 全局哪些电影最热门 |
| 统计个性化 | ItemCF | 用户喜欢过的电影和哪些电影相似 |
| 训练式个性化 | MF | 用户向量和电影向量的匹配关系 |

## DNN Ranker 排序模型报告解析

这个报告评估的是第五个模型：**DNN Ranker 排序模型**。

前面的 Popularity、ItemCF、MF、Two-Tower 都主要解决“召回”问题，也就是：

```text
从电影库里找出用户可能喜欢的一批候选电影。
```

DNN Ranker 解决的是“排序”问题，也就是：

```text
给定一个用户和一部候选电影，预测这个用户喜欢这部电影的概率，
然后按照概率从高到低排序。
```

对应代码在 [ranker_train.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/ranker_train.py)。

**这次 DNN Ranker 实验的核心流程**

```text
训练集正反馈
  ↓
构造用户-电影正样本
  ↓
为每个正样本随机采样负样本
  ↓
拼接 user embedding、item embedding、交叉向量和统计特征
  ↓
输入 DNN MLP
  ↓
用 BCE Loss 训练喜欢概率
  ↓
用 AUC / LogLoss 评估排序能力
  ↓
同时对全量电影打分并计算 Recall@20 / NDCG@20
```

当前 DNN Ranker 结果是：

| 指标 | 当前值 | 含义 |

| HitRate@20 | 0.106841 | 有多少用户的 Top 20 至少命中 1 个真实喜欢电影 |
| Precision@20 | 0.005342 | Top 20 推荐里有多少比例是真正相关电影 |
| NDCG@20 | 0.041302 | 命中电影是否排得靠前 |
| Coverage@20 | 0.525882 | 所有推荐结果覆盖了多少比例的电影库 |

| LogLoss | 0.138917 | 概率预测误差，越低越好 |

和已有模型对比：

| 模型 | Recall@20 | NDCG@20 | Coverage@20 | AUC | LogLoss |
|---|---:|---:|---:|---:|---:|
| Popularity | 0.067252 | 0.026149 | 0.045326 | - | - |
| ItemCF | 0.082160 | 0.032793 | 0.132372 | - | - |
| MF | 0.104025 | 0.039807 | 0.481844 | - | - |
| Two-Tower | 0.101872 | 0.040244 | 0.340973 | - | - |
| DNN-Ranker | 0.106841 | 0.041302 | 0.525882 | 0.899737 | 0.138917 |

**1. DNN Ranker 的模型架构**

当前 Ranker 不是 Transformer，也不是双塔结构，而是一个特征拼接后的 MLP 排序网络。

输入特征包括：

| 特征 | 含义 |
|---|---|
| `user_id embedding` | 用户 ID 的可训练向量 |
| `item_id embedding` | 电影 ID 的可训练向量 |
| `user_embedding * item_embedding` | 用户和电影向量的逐元素交叉 |
| 用户活跃度 | 用户训练集正反馈数量的归一化特征 |
| 电影热度 | 电影训练集正反馈数量的归一化特征 |
| 电影类别数量 | 电影 genre 数量的归一化特征 |
| 电影类别 multi-hot | MovieLens 的 18 个 genre 标签 |

当前模型结构是：

```text
输入特征
  ↓
Linear(input_dim -> 512)
  ↓
ReLU + Dropout(0.1)
  ↓
Linear(512 -> 256)
  ↓
ReLU + Dropout(0.1)
  ↓
Linear(256 -> 128)
  ↓
ReLU + Dropout(0.1)
  ↓
Linear(128 -> 1)
  ↓
输出 logit
```

如果按隐藏层来数：

```text
DNN Ranker 有 3 个隐藏层：512、256、128。
```

如果把最终输出层也算进去：

```text
一共有 4 个 Linear 层。
```

**2. DNN Ranker 的打分是什么**

Ranker 输出的是一个 logit。可以通过 sigmoid 转成概率：

$$
p = \sigma(logit)
$$

这个概率可以理解为：

```text
模型认为用户喜欢这部候选电影的概率。
```

在 TopK 评估时，不一定真的要把 logit 转成概率，因为 sigmoid 是单调函数。按 logit 从高到低排序和按概率从高到低排序，结果相同。

**3. DNN Ranker 怎么训练**

训练样本仍然来自 MovieLens 正反馈：

```text
rating >= 4 -> label = 1
```

负样本从用户没有出现过正反馈的电影中随机采样：

```text
未出现过正反馈的电影 -> label = 0
```

当前配置是：

```text
每个正样本训练负采样数 = 3
```

也就是说，每条正样本会配 3 条随机负样本。

训练目标是 `BCEWithLogitsLoss`：

```text
让正样本预测概率更接近 1，
让负样本预测概率更接近 0。
```

当前训练配置：

| 配置 | 当前值 | 含义 |
|---|---:|---|
| 设备 | mlu | 在公司寒武纪 MLU 容器中训练 |
| 用户数 | 6,038 | 用户 embedding 数量 |
| 电影数 | 3,883 | 电影 embedding 数量 |
| Genre 特征数 | 18 | MovieLens 电影类别数量 |
| Embedding Dim | 128 | 用户和电影 ID embedding 维度 |
| MLP Hidden Dims | 512 -> 256 -> 128 | 排序网络隐藏层 |
| Epochs | 8 | 完整遍历训练样本 8 轮 |
| Batch Size | 4,096 | 每批正样本数量 |
| 每个正样本训练负采样数 | 3 | 每条正样本配 3 条负样本 |
| 每个测试用户排序评估负采样数 | 100 | AUC/LogLoss 评估时每个用户采 100 个负样本 |
| Learning Rate | 0.001 | 优化器学习率 |
| Dropout | 0.1 | 防止过拟合 |
| Final Loss | 0.205959 | 最后一轮平均训练损失 |
| 训练耗时（秒） | 25.89 | MLU 上训练耗时 |
| 评估耗时（秒） | 14.10 | 排序评估和 TopK 评估耗时 |

**4. AUC 是怎么理解的**

AUC 衡量的是：

```text
随机拿一个正样本和一个负样本，
模型把正样本排在负样本前面的概率。
```

当前：

```text
AUC = 0.899737
```

可以理解为：大约 89.97% 的正负样本对中，模型给正样本的分数高于负样本。

这是排序模型里非常重要的指标。它不关心具体概率是否校准，只关心排序顺序是否正确。

**5. LogLoss 是怎么理解的**

LogLoss 衡量的是概率预测是否准确。

如果正样本预测概率很高，损失低；如果正样本预测概率很低，损失高。

如果负样本预测概率很低，损失低；如果负样本预测概率很高，损失高。

当前：

```text
LogLoss = 0.138917
```

这个值越低越好。它和 AUC 的区别是：

- AUC 更关心排序顺序。
- LogLoss 更关心概率是否靠谱。

**6. 为什么 Ranker 也有 Recall@20**

虽然 Ranker 本质是排序模型，但当前实验为了和前面模型对比，也让它对所有电影打分，然后取 Top 20。

这不是最标准的线上用法。更真实的企业推荐流程应该是：

```text
Two-Tower / ItemCF / MF 先召回 Top 200 或 Top 1000
  ↓
DNN Ranker 只对这些候选做精排
  ↓
取最终 Top 20
```

所以当前 Ranker 的 `Recall@20` 是一个“全量排序版”的离线结果。下一步更合理的实验，是实现“召回候选集重排”。

**7. 一句话总结**

DNN Ranker 的核心是：

```text
把用户、电影和统计特征拼接起来，
用多层 MLP 预测用户喜欢候选电影的概率，
再按概率对候选电影排序。
```

当前 `AUC = 0.899737`，说明它已经具备较强的正负样本区分能力；`Recall@20 = 0.106841`，也略高于 MF 和 Two-Tower。下一步应把它接到召回候选集之后，形成“召回 + 排序”的完整推荐 pipeline。



## Two-Tower + DNN Ranker 重排报告解析

这个报告评估的是第六个实验：**Two-Tower 召回 + DNN Ranker 重排**。

它不再是单独评估一个模型，而是把前面两个模型串成一个更接近企业推荐系统的两阶段 pipeline：

```text
第一阶段：Two-Tower 从全量电影中召回 Top 200 候选
第二阶段：DNN Ranker 对这 200 个候选重新打分
最终输出：Ranker 分数最高的 Top 20
```

对应代码在 [rerank_pipeline.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/rerank_pipeline.py)。

**这次两阶段实验的核心流程**

```text
训练 Two-Tower 召回模型
  ↓
训练 DNN Ranker 排序模型
  ↓
对每个测试用户用 Two-Tower 召回 Top 200 候选电影
  ↓
检查测试集真实电影是否进入候选集，计算 Candidate Recall@200
  ↓
用 DNN Ranker 对 200 个候选重新打分
  ↓
取重排后的 Top 20
  ↓
计算 Recall@20、NDCG@20、Coverage@20
```

当前两阶段 pipeline 结果是：

| 指标 | 当前值 | 含义 |
|---|---:|---|
| Candidate Recall@200 | 0.471757 | Two-Tower Top 200 候选中命中测试电影的比例 |
| Candidate HitRate@200 | 0.471757 | 有多少用户的 Top 200 候选至少命中测试电影 |
| Recall@20 | 0.108829 | 重排后 Top 20 命中用户真实喜欢电影的比例 |
| HitRate@20 | 0.108829 | 重排后 Top 20 至少命中 1 个测试电影的用户比例 |
| Precision@20 | 0.005441 | 重排后 Top 20 推荐中真正命中的比例 |
| NDCG@20 | 0.041877 | 重排后命中电影是否排得靠前 |
| Coverage@20 | 0.435230 | 重排后推荐结果覆盖的电影库比例 |
| AUC | 0.899737 | Ranker 区分正负样本的能力 |
| LogLoss | 0.138917 | Ranker 概率预测误差 |

和单模型结果对比：

| 模型 | Recall@20 | NDCG@20 | Coverage@20 |
|---|---:|---:|---:|
| MF | 0.104025 | 0.039807 | 0.481844 |
| Two-Tower | 0.101872 | 0.040244 | 0.340973 |
| DNN-Ranker | 0.106841 | 0.041302 | 0.525882 |
| TwoTower+DNN-Rerank | 0.108829 | 0.041877 | 0.435230 |

可以看到，两阶段重排后的 `Recall@20` 和 `NDCG@20` 是当前所有实验里最高的。

**1. Candidate Recall@200 是什么**

`Candidate Recall@200` 衡量的是：

```text
用户测试集中真实喜欢的电影，是否进入了 Two-Tower 召回的 Top 200 候选。
```

公式可以理解为：

$$
CandidateRecall@N = \frac{TopN 候选中命中的相关物品数}{用户真实相关物品数}
$$

当前：

```text
Candidate Recall@200 = 0.471757
```

表示大约 **47.18%** 的测试用户，其真实喜欢的下一部电影进入了 Two-Tower 的 Top 200 候选集。

这个指标非常重要，因为它决定了排序阶段的上限：

```text
如果真实电影没有进入候选集，
Ranker 再强也不可能把它排进最终 Top 20。
```

所以召回阶段的任务不是直接给出最终排序，而是尽可能把潜在相关物品放进候选池。

**2. 为什么要先召回再排序**

真实推荐系统里，物品数量可能是百万级、千万级甚至更大。如果直接用复杂 Ranker 给所有物品打分，成本太高。

所以通常采用两阶段：

```text
召回模型：快，负责从海量物品中找几百到几千个候选
排序模型：慢但更准，负责对候选做精细打分
```

在当前 MovieLens 实验里，电影总数只有 3,883，直接全量排序也能跑。但为了模拟企业推荐架构，我们仍然做成：

```text
3,883 部电影
  ↓ Two-Tower
200 个候选
  ↓ DNN Ranker
20 个最终推荐
```

**3. 两阶段 pipeline 是怎么打分的**

第一阶段 Two-Tower 打分：

```text
score_recall(user, item) = 用户塔向量 和 物品塔向量 的相似度
```

它只用于选出 Top 200 候选。

第二阶段 DNN Ranker 打分：

```text
score_rank(user, item) = MLP(user 特征, item 特征, 交叉特征)
```

它只在这 200 个候选里重新排序。

最终推荐结果来自 Ranker：

```text
先按 Two-Tower 分数取 Top 200
再按 Ranker 分数取 Top 20
```

**4. 为什么重排后 Recall@20 能提升**

Two-Tower 更擅长“粗略找相似候选”，Ranker 更擅长“精细判断候选顺序”。

如果 Two-Tower 已经把真实喜欢的电影召回到了 Top 200，Ranker 就有机会把它往前排到 Top 20。

当前结果：

```text
DNN-Ranker 全量排序 Recall@20 = 0.106841
TwoTower+DNN-Rerank Recall@20 = 0.108829
```

提升不算巨大，但方向是对的，说明两阶段 pipeline 已经跑通，并且当前候选集重排带来了增益。

**5. 为什么 Coverage@20 下降了**

DNN-Ranker 全量排序的 Coverage@20 是：

```text
0.525882
```

两阶段重排后的 Coverage@20 是：

```text
0.435230
```

这是正常现象，因为重排阶段只能从 Two-Tower 召回的 Top 200 候选里选最终结果。候选池本身会限制最终推荐覆盖范围。

换句话说：

```text
全量 Ranker 可以从全部 3,883 部电影里选。
两阶段 Ranker 只能从每个用户的 200 部候选电影里选。
```

所以它可能牺牲一部分覆盖率，但换来更接近真实系统的计算流程。

**6. 当前训练和评估设置**

| 配置 | 当前值 | 含义 |
|---|---:|---|
| 设备 | mlu | 在公司寒武纪 MLU 容器中训练 |
| 用户数 | 6,038 | 用户数量 |
| 电影数 | 3,883 | 电影数量 |
| Two-Tower Embedding Dim | 128 | 双塔 ID embedding 维度 |
| Two-Tower Hidden Dim | 256 | 双塔隐藏层维度 |
| Two-Tower Tower Dim | 128 | 双塔输出向量维度 |
| Two-Tower Epochs | 10 | 双塔训练轮数 |
| Two-Tower Final Loss | 0.313643 | 双塔最终训练损失 |
| Two-Tower 训练耗时（秒） | 23.38 | 双塔训练耗时 |

**7. 这个实验的企业级意义**

这个实验比单独训练 MF、Two-Tower 或 Ranker 更有工程价值，因为它已经体现了推荐系统的典型分层架构：

```text
召回：快速缩小候选范围
排序：对候选做精细打分
评估：同时看候选召回率、最终 TopK 指标和排序指标
```

简历或面试中可以这样描述：

```text
构建 Two-Tower + DNN Ranker 两阶段推荐 pipeline，
先通过双塔向量召回生成 Top 200 候选，
再使用融合 ID、热度、类别和交叉特征的 DNN Ranker 重排 Top 20，
并基于 Candidate Recall@K、Recall@K、NDCG@K、AUC、LogLoss 完成离线评测。
```

**8. 一句话总结**

Two-Tower + DNN Ranker 重排的核心是：

```text
先用快模型召回候选，
再用强模型重排候选，
最终输出更准确的 Top 20 推荐。
```

当前 `Candidate Recall@200 = 0.471757`，说明 Two-Tower 能把约 47.18% 用户的真实测试电影召回进候选集；重排后 `Recall@20 = 0.108829`、`NDCG@20 = 0.041877`，是当前 MovieLens 实验中最好的 TopK 结果。



## Candidate K 消融实验解析

这个实验回答一个很关键的问题：

```text
Two-Tower 到底应该给 Ranker 召回多少候选？
```

候选数太小，真实相关电影可能进不了候选池，Ranker 没机会把它排上去。

候选数太大，虽然召回上限更高，但会混入更多噪声候选，Ranker 的排序难度和计算耗时都会增加。

对应代码在 [rerank_ablation.py](E:/programs/x-algorithm/experiments/movielens_recall/scripts/rerank_ablation.py)。

**这次消融实验怎么做**

为了保证对比公平，脚本不是分别训练 4 次模型，而是：

```text
训练一次 Two-Tower
  ↓
训练一次 DNN Ranker
  ↓
用 Two-Tower 生成最大 Top 500 候选
  ↓
分别截断成 Top 50、Top 100、Top 200、Top 500
  ↓
用同一个 Ranker 重排
  ↓
对比每个 candidate_k 的最终 Top20 指标
```

这样不同候选数之间的差异主要来自候选池大小，而不是模型重新训练带来的随机波动。

**当前消融结果**

| Candidate K | Candidate Recall | Recall@20 | NDCG@20 | Coverage@20 | 重排耗时（秒） |
|---:|---:|---:|---:|---:|---:|
| 50 | 0.198277 | 0.109988 | 0.042451 | 0.366469 | 4.19 |
| 100 | 0.315388 | 0.108332 | 0.041980 | 0.399176 | 4.37 |
| 200 | 0.471757 | 0.108829 | 0.041877 | 0.435230 | 4.63 |
| 500 | 0.733477 | 0.107172 | 0.041467 | 0.477981 | 5.71 |

**1. 为什么 Candidate Recall 会越来越高**

`Candidate Recall` 衡量的是：

```text
测试集中真实喜欢的电影是否进入 Two-Tower 候选池。
```

候选池越大，越容易包含真实电影。

所以结果是单调上升的：

```text
Top 50  Candidate Recall = 0.198277
Top 100 Candidate Recall = 0.315388
Top 200 Candidate Recall = 0.471757
Top 500 Candidate Recall = 0.733477
```

这很好理解：给 Ranker 的候选越多，真实答案进入候选池的概率越高。

**2. 为什么最终 Recall@20 不是越来越高**

最终 `Recall@20` 看的是：

```text
Ranker 重排之后，真实电影是否进入最终 Top 20。
```

候选池变大后，Ranker 能看到更多可能正确的电影，但也会看到更多干扰项。

当前结果里：

```text
candidate_k=50  Recall@20 = 0.109988
candidate_k=500 Recall@20 = 0.107172
```

虽然 Top 500 的候选召回上限更高，但最终 Top20 反而更低。说明当前 Ranker 在更大的候选池里没有稳定地把真实电影排进前 20，更多噪声候选会干扰最终排序。

**3. 为什么 candidate_k=50 当前最好**

`candidate_k=50` 的特点是：

- 候选池更小，排序任务更简单。
- Two-Tower 前 50 候选质量相对更高。
- Ranker 在较干净的候选集合里更容易选出前 20。

所以它取得了当前最高的：

```text
Recall@20 = 0.109988
NDCG@20 = 0.042451
```

这不代表所有场景都应该用 50，而是说明当前 MovieLens 1M、当前 Two-Tower 和 Ranker 配置下，Top 50 是更好的折中点。

**4. Coverage 和耗时怎么理解**

Coverage 随候选数增大整体上升：

```text
Top 50  Coverage@20 = 0.366469
Top 500 Coverage@20 = 0.477981
```

说明更大的候选池会让最终推荐覆盖更多电影。

重排耗时也随候选数增加：

```text
Top 50  重排耗时 = 4.19 秒
Top 500 重排耗时 = 5.71 秒
```

在 MovieLens 这种小数据集上差异不大，但在真实业务里，如果候选从 500 增加到 5000，Ranker 计算成本会明显上升。

**5. 这个实验说明了什么**

这个消融实验说明，推荐系统不是简单地“候选越多越好”。

更准确的说法是：

```text
召回阶段要提供足够高的上限，
排序阶段要能在候选池中把真正相关的物品排到前面，
候选数量需要在效果、覆盖和计算成本之间折中。
```

**6. 一句话总结**

当前最佳候选数是：

```text
candidate_k = 50
```

它取得了：

```text
Recall@20 = 0.109988
NDCG@20 = 0.042451
```

这是当前 MovieLens 阶段最好的 Top20 推荐结果，也说明两阶段推荐 pipeline 已经具备进一步做工程优化和数据集迁移的基础。
