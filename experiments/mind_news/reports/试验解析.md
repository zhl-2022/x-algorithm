# MIND 新闻推荐试验解析

这个文档用于记录 `experiments/mind_news/` 阶段的全部学习路线、数据处理方案、
当前已完成实验、后续模型设计和每一步应该如何理解。目标是让没有推荐系统背景的人，
也能看懂这个新闻推荐实验为什么要做、怎么做、指标怎么看、下一步做什么。

## 1. 这个阶段为什么要做 MIND

MovieLens 阶段已经跑通了推荐系统最小闭环：

```text
数据处理
  ↓
Popularity / ItemCF / MF / Two-Tower
  ↓
DNN Ranker
  ↓
Two-Tower 召回 + Ranker 重排
  ↓
candidate_k 消融实验
```

但 MovieLens 主要是电影 ID、评分和类别，内容信息很弱。它适合入门，但不太像
X/Twitter、新闻流、短视频流这类真实内容推荐系统。

MIND 是新闻推荐数据集，它多了这些关键内容：

| 内容 | 为什么重要 |
|---|---|
| 新闻标题 | 可以训练新闻内容表示，类似帖子文本 |
| 新闻摘要 | 可以提供更丰富的语义信息 |
| 新闻类别 | 可以做类别偏好、冷启动和排序特征 |
| 用户点击历史 | 可以建模用户兴趣序列 |
| 曝光点击标签 | 可以训练排序模型，评估 CTR 预测能力 |

所以 MIND 阶段的目标是从“ID 推荐”升级到“内容理解 + 用户行为推荐”。

## 2. 当前数据源说明

原计划使用 MIND-small 官方 Azure Blob 数据：

```text
MINDsmall_train.zip
MINDsmall_dev.zip
```

但当前官方 Blob 下载返回：

```text
Public access is not permitted
```

为了继续推进实验，当前使用可访问的 RecZoo `MIND_small_x1` 镜像。这个镜像不是官方
原始 `news.tsv + behaviors.tsv` 格式，而是已经将曝光展开成逐条样本：

```text
user_id, news_id, click, news_his, cat, sub_cat, title_entities, abstract_entities
```

这对后续训练排序模型和构造召回样本更方便。

$$

## 5. 当前数据文件怎么理解

### `news_metadata.csv`

这个文件记录新闻内容元数据：

| 字段 | 含义 |
|---|---|
| `news_id` | 新闻 ID |
| `category` | 一级类别 |
| `subcategory` | 二级类别 |
| `title` | 新闻标题 |
| `abstract` | 新闻摘要 |
| `title_entities` | 标题实体 |
| `abstract_entities` | 摘要实体 |
| `title_word_count` | 标题词数 |
| `abstract_word_count` | 摘要词数 |

后续新闻塔会用 `title`、`abstract`、`category`、`subcategory` 生成新闻向量。

### `train_sample.csv` 和 `valid_sample.csv`

这两个文件是从 RecZoo 大 CSV 中抽取的前 100,000 行样本，用于快速调试模型。

| 字段 | 含义 |
|---|---|
| `user_id` | 用户 ID |
| `news_id` | 当前曝光新闻 |
| `click` | 是否点击，1 表示点击，0 表示未点击 |
| `hour` | 曝光小时 |
| `cat` | 当前新闻一级类别 |
| `sub_cat` | 当前新闻二级类别 |
| `news_his` | 用户历史点击新闻序列 |
| `cat_his` | 用户历史点击新闻类别序列 |
| `subcat_his` | 用户历史点击新闻二级类别序列 |

后续训练时：

```text
click = 1 作为正样本
click = 0 作为负样本
```

## 6. 和 MovieLens 阶段有什么不同

| 对比点 | MovieLens | MIND |
|---|---|---|
| 推荐对象 | 电影 | 新闻 |
| 反馈类型 | 评分转正反馈 | 曝光点击标签 |
| 正样本定义 | `rating >= 4` | `click = 1` |
| 负样本来源 | 随机采样未正反馈电影 | 原始曝光未点击新闻 |
| 内容特征 | 电影类别较弱 | 标题、摘要、类别、实体 |
| 用户历史 | 喜欢过的电影 ID | 点击过的新闻 ID 序列 |
| 更接近业务 | 一般 | 更接近信息流推荐 |

MIND 的价值在于：它可以训练内容感知推荐模型，而不是只学 ID embedding。

## 8. 下一步具体做什么

下一步最合理的是做 MIND 的最低 baseline：

```text
MIND Popularity baseline
```

它的逻辑是：

1. 在训练集中统计每篇新闻被点击的次数。
2. 对验证集每个用户，推荐训练集中点击次数最高的 TopK 新闻。
3. 过滤用户历史里已经点击过的新闻。
4. 用验证集 `click=1` 的新闻作为真实相关物品。
5. 计算 `Recall@K`、`HitRate@K`、`Precision@K`、`NDCG@K`、`Coverage@K`。

这个 baseline 的作用和 MovieLens 的 Popularity 一样：它不是为了强，而是为了建立最低基准。

## 9. 后续模型设计

### 9.1 内容感知 Two-Tower

MIND 的 Two-Tower 可以设计成：

```text
用户塔：
  user_id embedding
  + 用户历史点击新闻 embedding 均值
  + 用户历史类别 embedding 均值

新闻塔：
  news_id embedding
  + category embedding
  + subcategory embedding
  + title text embedding
  + abstract text embedding

打分：
  用户向量 · 新闻向量
```

这比 MovieLens 的双塔更接近真实内容推荐。

### 9.2 DNN Ranker

Ranker 可以融合：

| 特征 | 说明 |
|---|---|
| 用户 ID embedding | 用户个性化偏好 |
| 新闻 ID embedding | 新闻本身热度和内容偏好 |
| 类别 embedding | 用户类别兴趣 |
| 标题/摘要 embedding | 新闻语义 |
| 用户历史长度 | 用户活跃度 |
| 新闻点击热度 | 新闻流行度 |
| 曝光时间 | 时间特征 |
| 用户-新闻交叉向量 | 个性化匹配 |

输出是：

```text
用户点击该新闻的概率
```

评估指标是：

```text
AUC
LogLoss
MRR
NDCG@5
NDCG@10
```

## 10. 关于“Grok 基座模型”的理解

这里不建议从零训练一个类似 Grok 的大语言模型基座。原因很直接：

1. 推荐系统目标不是生成文本，而是做用户-内容匹配。
2. 从零训练 LLM 成本极高，和当前项目目标不匹配。
3. 简历里更有价值的是推荐系统闭环，而不是泛泛说训练了一个大模型。

更合理的方向是：

```text
小型内容推荐基座表征模型
```

也就是：

```text
文本编码器理解新闻内容
用户塔建模用户历史兴趣
召回模型学习用户-内容向量空间
排序模型学习点击概率
```

这条路线更贴近 X/Twitter 推荐系统，也更适合当前两张 80GB MLU 的学习目标。

## 11. 当前阶段一句话总结

MIND 阶段已经完成数据入口建设。接下来要做的是：

```text
先建立新闻推荐 baseline，
再逐步把 MovieLens 阶段的召回、排序、重排 pipeline 迁移到新闻推荐上，
最后加入文本内容编码，形成内容理解型推荐系统。
```
