# MIND 新闻推荐试验解析

这个文档用于记录 `experiments/mind_news/` 阶段的全部学习路线、数据处理方案、
当前已完成实验、后续模型设计和每一步应该如何理解。目标是让没有推荐系统背景的人，
也能看懂这个新闻推荐实验为什么要做、怎么做、指标怎么看、下一步做什么。

## 1. 这个阶段为什么要做 MIND

MovieLens 阶段已经跑通了推荐系统最小闭环：

```text
数据处理
  ↓
Popularity / ItemCF / MF / Two-Tower
  ↓
DNN Ranker
  ↓
Two-Tower 召回 + Ranker 重排
  ↓
candidate_k 消融实验
```

但 MovieLens 主要是电影 ID、评分和类别，内容信息很弱。它适合入门，但不太像
X/Twitter、新闻流、短视频流这类真实内容推荐系统。

MIND 是新闻推荐数据集，它多了这些关键内容：

| 内容 | 为什么重要 |
|---|---|
| 新闻标题 | 可以训练新闻内容表示，类似帖子文本 |
| 新闻摘要 | 可以提供更丰富的语义信息 |
| 新闻类别 | 可以做类别偏好、冷启动和排序特征 |
| 用户点击历史 | 可以建模用户兴趣序列 |
| 曝光点击标签 | 可以训练排序模型，评估 CTR 预测能力 |

所以 MIND 阶段的目标是从“ID 推荐”升级到“内容理解 + 用户行为推荐”。


这对后续训练排序模型和构造召回样本更方便。

$$

$$

CTR 只有约 4%，说明正负样本非常不平衡。这更接近真实推荐系统：绝大多数曝光不会被点击。

### `news_metadata.csv`

这个文件记录新闻内容元数据：

| 字段 | 含义 |
|---|---|
| `news_id` | 新闻 ID |
| `category` | 一级类别 |
| `subcategory` | 二级类别 |

| `abstract_entities` | 摘要实体 |

后续新闻塔会用 `title`、`abstract`、`category`、`subcategory` 生成新闻向量。

### `train_sample.csv` 和 `valid_sample.csv`

这两个文件是从 RecZoo 大 CSV 中抽取的前 100,000 行样本，用于快速调试模型。

| 字段 | 含义 |
|---|---|
| `user_id` | 用户 ID |
| `news_id` | 当前曝光新闻 |
| `click` | 是否点击，1 表示点击，0 表示未点击 |
| `hour` | 曝光小时 |
| `cat` | 当前新闻一级类别 |
| `sub_cat` | 当前新闻二级类别 |
| `news_his` | 用户历史点击新闻序列 |
| `cat_his` | 用户历史点击新闻类别序列 |
| `subcat_his` | 用户历史点击新闻二级类别序列 |

后续训练时：

```text
click = 1 作为正样本
click = 0 作为负样本
```

## 6. 和 MovieLens 阶段有什么不同

| 对比点 | MovieLens | MIND |
|---|---|---|
| 推荐对象 | 电影 | 新闻 |
| 反馈类型 | 评分转正反馈 | 曝光点击标签 |
| 正样本定义 | `rating >= 4` | `click = 1` |
| 负样本来源 | 随机采样未正反馈电影 | 原始曝光未点击新闻 |
| 内容特征 | 电影类别较弱 | 标题、摘要、类别、实体 |
| 用户历史 | 喜欢过的电影 ID | 点击过的新闻 ID 序列 |
| 更接近业务 | 一般 | 更接近信息流推荐 |

MIND 的价值在于：它可以训练内容感知推荐模型，而不是只学 ID embedding。

## 8. 已完成：MIND Popularity baseline

MIND 的第一个 baseline 已经完成：

```text
训练集统计新闻点击热度
  ↓
验证集每个 impression 内部按热度排序
  ↓
计算 AUC / MRR / NDCG@5 / NDCG@10 / HitRate / Coverage
```

### 8.1 这个模型用了什么信息

Popularity baseline 只使用一个信息：

```text
新闻在训练集中被点击了多少次
```

它没有使用：

- `user_id`：不知道当前用户是谁。
- `news_his`：不知道用户过去看过什么。
- `title` / `abstract`：不理解新闻文本内容。
- `category` / `subcategory`：不理解新闻类别。

所以它是一个非个性化模型。它的价值不是效果强，而是给后续模型提供最低对照线。

### 8.2 热度分数怎么计算

每篇新闻的主分数是训练集点击次数：

$$
score(news) = click\_count(news)
$$

如果两篇新闻点击数完全相同，脚本会使用一个极小的曝光次数项打破并列：

$$
score(news) = click\_count(news) + \lambda \cdot log(1 + exposure\_count(news))
$$

本次实验使用：

```text
lambda = 1e-6
```

这个曝光项很小，不会改变“点击数越高越热门”的主逻辑，只是让并列新闻有稳定顺序。

### 8.3 它是怎么做推荐的

RecZoo 版 MIND 已经把一次曝光展开成多行样本。相同的 `imp_id + user_id` 表示同一次推荐曝光里的候选新闻列表。例如：

```text
imp_id=1, user_id=U13740, news_id=N55689, click=1
imp_id=1, user_id=U13740, news_id=N35729, click=0
imp_id=1, user_id=U13740, news_id=N43111, click=0
```

脚本不会从 65,238 篇新闻里重新召回 TopK，而是在这一次曝光的候选新闻内部排序：

```text
候选新闻 A：训练集中点击 4316 次
候选新闻 B：训练集中点击 3346 次
候选新闻 C：训练集中点击 10 次

排序结果：A > B > C
```

然后看真实点击的新闻是否被排在前面。

这和 MovieLens 阶段不同：

| 阶段 | 评估方式 |
|---|---|
| MovieLens | 从全量电影里推荐 TopK，再看是否命中测试集中用户喜欢的电影 |
| MIND | 在一次真实曝光候选列表里排序，再看点击新闻是否排得靠前 |

MIND 这种方式更像真实信息流排序评测。

### 8.4 输入和输出

| 类型 | 路径 | 说明 |
|---|---|---|
| 训练输入 | `data/raw/MIND_small_x1/train.csv` | 用来统计新闻点击次数 |
| 验证输入 | `data/raw/MIND_small_x1/valid.csv` | 用来按曝光组评估排序效果 |
| 新闻元数据 | `data/processed/news_metadata.csv` | 用来在报告里展示新闻标题和类别 |
| 训练脚本 | `scripts/popularity_baseline.py` | Popularity baseline 实现 |
| 报告 | `reports/baseline_report.md` | 面向阅读的实验报告 |
| 结果表 | `outputs/popularity_results.csv` | 机器可读指标 |
| 汇总表 | `outputs/experiment_results.csv` | 当前 MIND 阶段统一结果表 |

### 8.5 本次实验指标

| 指标 | 数值 | 怎么理解 |
|---|---:|---|
| AUC | 0.522252 | 随机拿一条点击新闻和一条未点击新闻，热门模型把点击新闻排更高的概率 |
| MRR | 0.266074 | 每个曝光列表中，第一条被点击新闻越靠前越高 |
| NDCG@5 | 0.247261 | Top5 中点击新闻越靠前越高 |
| NDCG@10 | 0.308465 | Top10 中点击新闻越靠前越高 |
| HitRate@5 | 0.423433 | Top5 至少出现一条点击新闻的曝光组比例 |
| HitRate@10 | 0.611986 | Top10 至少出现一条点击新闻的曝光组比例 |
| Coverage@5 | 0.017061 | 所有 Top5 推荐覆盖到的不同新闻占总新闻数比例 |
| Coverage@10 | 0.026380 | 所有 Top10 推荐覆盖到的不同新闻占总新闻数比例 |

### 8.6 结论

Popularity baseline 的 AUC 只有 `0.522252`，略高于随机排序的 `0.5`。这说明热门程度对新闻点击有一点帮助，
但帮助有限。原因也很好理解：用户是否点击新闻，不只取决于新闻热不热门，还取决于用户兴趣、新闻类别、标题内容和当时上下文。

这个结果给后续模型设定了明确目标：

```text
后续模型必须超过 AUC=0.522252、MRR=0.266074、NDCG@10=0.308465，
才说明个性化特征或内容特征真的带来了收益。
```

## 9. 下一步具体做什么

当前已经完成 MIND 阶段的本地可复现实验闭环。下一步建议做：

```text
MIND 全量神经模型 + MLU 性能实验
```

原因是本地 CPU 已经验证了模型链路，继续在本地堆训练轮数价值有限；更有价值的是把 DNN Ranker
和 ContentTwoTower 迁移到 MLU 上跑更大训练集。

### 9.1 当前全部实验结果

| 模型 | 范围 | AUC | MRR | NDCG@5 | NDCG@10 | HitRate@10 | 怎么理解 |
|---|---|---:|---:|---:|---:|---:|---|
| Popularity | full | 0.522252 | 0.266074 | 0.247261 | 0.308465 | 0.611986 | 只看新闻全局点击热度 |
| Category | full | 0.588720 | 0.291866 | 0.274974 | 0.338507 | 0.657152 | 加入类别 CTR 和用户历史类别偏好 |
| Popularity-Sample | sample | 0.515232 | 0.265642 | 0.247214 | 0.308516 | 0.605566 | 在样本验证集上的热门基线 |
| Category-Sample | sample | 0.591423 | 0.289128 | 0.274184 | 0.335716 | 0.647866 | 在样本验证集上的类别基线 |
| DNNRanker | sample | 0.546448 | 0.266533 | 0.238774 | 0.311597 | 0.633766 | MLP 学习点击概率 |
| ContentTwoTower | sample | 0.585406 | 0.289335 | 0.276951 | 0.339823 | 0.669759 | 用户历史和新闻内容进入双塔向量 |
| TwoTower+DNN-Rerank | sample | 0.555439 | 0.273330 | 0.243986 | 0.319061 | 0.650093 | 双塔先筛候选，DNN 再重排 |

### 9.2 为什么有 full 和 sample 两种范围

| 范围 | 含义 | 当前使用场景 |
|---|---|---|
| `full` | 使用全量 5,843,444 条训练样本和 2,740,998 条验证样本 | Popularity、Category 这类统计模型 |
| `sample` | 使用 `train_sample.csv` 和 `valid_sample.csv` 各 100,000 行 | DNNRanker、ContentTwoTower、Pipeline |

这样设计是为了兼顾两件事：

1. 统计模型足够快，可以直接全量跑。
2. 神经模型先验证代码链路和指标闭环，后续再迁移到 MLU 做全量训练。

### 9.3 每个模型做了什么

| 模型 | 输入 | 输出 | 核心思想 |
|---|---|---|---|
| Popularity | `news_id`、训练集点击数 | 新闻热度分数 | 热门新闻更可能被点 |
| Category | 新闻热度、`cat`、`sub_cat`、`cat_his`、`subcat_his` | 类别增强分数 | 用户更可能点历史偏好类别里的新闻 |
| DNNRanker | 用户、新闻、类别、小时、热度、历史偏好、内容统计 | 点击概率 | 用 MLP 学非线性特征组合 |
| ContentTwoTower | 用户历史类别、新闻类别、新闻实体、内容统计 | 用户向量和新闻向量点积分数 | 学用户兴趣和新闻内容的向量匹配 |
| TwoTower+DNN-Rerank | Two-Tower 分数、DNNRanker 分数 | 重排后列表 | 先筛候选，再排序 |

### 9.4 当前结果怎么解释

全量 `Category` 的 `AUC=0.588720`，明显高于 `Popularity` 的 `AUC=0.522252`。
这说明新闻类别和用户历史类别偏好确实有用。

样本口径下，`ContentTwoTower` 的 `NDCG@10=0.339823`，高于 `DNNRanker` 的 `0.311597`。
这说明第一版内容双塔已经能从用户历史类别和新闻内容特征中学到一定排序能力。

`TwoTower+DNN-Rerank` 当前没有超过单独的 `ContentTwoTower`。这不是坏事，说明第一版 Ranker
还不够强，后续应该加强 Ranker 特征、训练样本规模和 `candidate_k` 消融。

### 9.5 下一步工程放大

后续建议：

1. 把 `run_all_experiments.py` 的神经模型训练行数从 100,000 提升到全量或百万级。
2. 在 MLU 容器里运行 DNNRanker 和 ContentTwoTower，记录训练耗时、显存和吞吐。
3. 对 pipeline 做 `candidate_k=10/20/50/100` 消融。
4. 将标题和摘要从长度/实体特征升级为文本 encoder 向量。

## 10. 后续模型设计

### 10.1 内容感知 Two-Tower

MIND 的 Two-Tower 可以设计成：

```text
用户塔：
  user_id embedding
  + 用户历史点击新闻 embedding 均值
  + 用户历史类别 embedding 均值

新闻塔：
  news_id embedding
  + category embedding
  + subcategory embedding
  + title text embedding
  + abstract text embedding

打分：
  用户向量 · 新闻向量
```

这比 MovieLens 的双塔更接近真实内容推荐。

### 10.2 DNN Ranker

Ranker 可以融合：

| 特征 | 说明 |
|---|---|
| 用户 ID embedding | 用户个性化偏好 |
| 新闻 ID embedding | 新闻本身热度和内容偏好 |
| 类别 embedding | 用户类别兴趣 |
| 标题/摘要 embedding | 新闻语义 |
| 用户历史长度 | 用户活跃度 |
| 新闻点击热度 | 新闻流行度 |
| 曝光时间 | 时间特征 |
| 用户-新闻交叉向量 | 个性化匹配 |

输出是：

```text
用户点击该新闻的概率
```

评估指标是：

```text
AUC
LogLoss
MRR
NDCG@5
NDCG@10
```

## 11. 关于“Grok 基座模型”的理解

这里不建议从零训练一个类似 Grok 的大语言模型基座。原因很直接：

1. 推荐系统目标不是生成文本，而是做用户-内容匹配。
2. 从零训练 LLM 成本极高，和当前项目目标不匹配。
3. 简历里更有价值的是推荐系统闭环，而不是泛泛说训练了一个大模型。

更合理的方向是：

```text
小型内容推荐基座表征模型
```

也就是：

```text
文本编码器理解新闻内容
用户塔建模用户历史兴趣
召回模型学习用户-内容向量空间
排序模型学习点击概率
```

这条路线更贴近 X/Twitter 推荐系统，也更适合当前两张 80GB MLU 的学习目标。
