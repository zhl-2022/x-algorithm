# KuaiRec 短视频推荐试验方案

## 1. 为什么第三批数据集选择 KuaiRec

MovieLens 和 MIND 已经分别覆盖了电影推荐和新闻推荐：

| 数据集 | 场景 | 已学习到的重点 |
|---|---|---|
| MovieLens 1M | 用户给电影评分 | 推荐系统离线闭环、召回、排序、TopK 评测 |
| MIND-small | 用户点击新闻 | 曝光候选列表、新闻内容特征、内容感知双塔、MLU 放大训练 |
| KuaiRec | 用户观看短视频 | 观看反馈、完播率、视频类别、短视频信息流推荐 |

KuaiRec 更接近短视频/信息流业务。短视频推荐不只是判断“点没点击”，还要关注用户是否看完、
是否重复观看、视频时长、类别兴趣和内容文本描述。这些信号比 MovieLens 的评分更接近真实业务日志。

## 2. 本阶段要解决什么问题

本阶段的核心问题是：

> 给定用户历史观看行为和短视频内容特征，模型能否把用户更可能喜欢的视频排到更前面？

拆成工程步骤就是：

1. 读取 KuaiRec 原始交互数据。
2. 明确每一行交互代表什么：用户、视频、观看时间、视频时长、完播率、时间戳等。
3. 把连续反馈转换成推荐模型可学习的标签。
4. 构造训练集、验证集和测试集。
5. 先跑简单 baseline，再跑可训练模型。
6. 用统一指标判断模型是否变好。
7. 把训练过程迁移到 MLU，并记录性能。

## 4. 标签怎么定义

短视频推荐里，标签不是天然的 1 分到 5 分评分，而是要从行为日志中构造。

### 4.1 首选正反馈定义

如果数据中存在 `watch_ratio`，优先使用：

$$
positive = \mathbb{1}(watch\_ratio \ge 1.0)
$$

含义是：用户观看时长达到或超过视频时长，认为用户对这个视频有明显兴趣。

### 4.2 备选正反馈定义

如果完播门槛过严，可以做消融：

| 标签策略 | 含义 | 适用情况 |
|---|---|---|
| `watch_ratio >= 1.0` | 完播或重复观看 | 标签更强，但正样本可能更少 |
| `watch_ratio >= 0.8` | 观看 80% 以上 | 更宽松，适合样本稀疏时 |
| `is_click = 1` | 发生点击/播放 | 如果字段存在，可作为点击预测标签 |
| `watch_time / duration >= 0.8` | 手动计算完播率 | 如果没有 `watch_ratio` 但有观看时长和视频时长 |

### 4.3 为什么不能随便定义标签

标签定义会直接影响模型学到什么：

- 用 `is_click`，模型学的是“用户会不会点开”。
- 用 `watch_ratio >= 1.0`，模型学的是“用户会不会看完”。
- 用 `watch_ratio` 回归，模型学的是“用户会看多久”。

本阶段先做二分类推荐，等 baseline 跑通后再扩展到观看时长回归或多任务学习。

## 5. 数据切分方式

优先使用按用户时间序列切分：

1. 对每个用户的交互按时间排序。
2. 最后一条或最后 20% 作为测试集。
3. 倒数第二条或中间一段作为验证集。
4. 更早的历史作为训练集。

这样更接近线上推荐：模型只能用过去行为预测未来兴趣。

如果某些交互表没有时间字段，则先做随机切分，并在报告里明确说明这是简化实验。

## 6. 第一轮模型清单

| 顺序 | 模型 | 作用 | 是否需要 MLU |
|---:|---|---|---|
| 1 | Popularity | 推荐最热门视频，作为最低基线 | 否 |
| 2 | Category baseline | 根据用户历史类别偏好推荐 | 否 |
| 3 | ItemCF | 根据用户看过的视频找相似视频 | 否 |
| 4 | MF | 学习用户和视频 embedding | 是 |
| 5 | Two-Tower | 用户塔和视频塔向量召回 | 是 |
| 6 | DNN Ranker | 融合用户、视频、类别、热度等特征排序 | 是 |
| 7 | Two-Tower + Ranker | 先召回再重排，模拟企业 pipeline | 是 |

## 7. 评测指标

| 指标 | 解释 | 用在什么阶段 |
|---|---|---|
| `Recall@K` | 用户真实喜欢的视频有多少被 TopK 推荐命中 | 召回、最终 TopK |
| `HitRate@K` | 用户 TopK 中是否至少命中一个正样本 | 召回、最终 TopK |
| `NDCG@K` | 命中越靠前分数越高 | 召回、排序、最终 TopK |
| `Precision@K` | TopK 中有多少比例是真正正样本 | 最终推荐列表 |
| `Coverage@K` | 推荐列表覆盖了多少不同视频 | 多样性和长尾 |
| `AUC` | 正样本得分是否普遍高于负样本 | CTR/完播排序 |
| `LogLoss` | 概率预测是否校准 | Ranker |

## 8. MLU 训练记录口径

每次 MLU 实验至少记录：

| 字段 | 说明 |
|---|---|
| 数据集 | 例如 `KuaiRec` |
| 模型 | MF、Two-Tower、DNN Ranker 等 |
| 训练样本数 | 实际用于训练的样本行数 |
| 验证样本数 | 实际用于评估的样本行数 |
| batch size | 每步训练样本数 |
| embedding dim | embedding 维度 |
| epoch | 训练轮数 |
| 训练耗时 | 秒或分钟 |
| 吞吐 | samples/s |
| 峰值显存 | 从 `cnmon` 采样 |
| 指标 | `AUC`、`NDCG@10`、`HitRate@10` 等 |

## 10. 第一轮训练结果

当前已完成 `small_matrix.csv` 第一轮完整训练，训练设置如下：

| 项目 | 数值 |
|---|---:|
| 数据源 | `small_matrix.csv` |
| 用户数 | 1,411 |
| 视频数 | 3,327 |
| 训练交互 | 3,595,097 |
| 验证交互 | 449,374 |
| 测试交互 | 450,107 |
| 训练正样本 | 1,177,823 |
| 测试正样本 | 153,264 |
| 神经模型训练样本 | 1,200,000 |
| Epochs | 2 |
| Batch size | 8,192 |

| 模型 | Recall@20 | HitRate@20 | NDCG@20 | AUC | 结论 |
|---|---:|---:|---:|---:|---|
| Popularity | 0.011655 | 0.631915 | 0.055724 | 0.480085 | 只看全局热门，作为最低基线 |
| Category | 0.017828 | 0.827660 | 0.083583 | 0.534036 | 类别偏好有效，明显强于热度 |
| ItemCF | 0.019690 | 0.778723 | 0.097525 | 0.478955 | 个性化协同过滤继续提升 TopK |
| MF | 0.025962 | 0.892908 | 0.127836 | 0.683895 | 可训练 embedding 明显提升召回 |
| Two-Tower | 0.026051 | 0.901418 | 0.143577 | 0.549570 | 当前 `NDCG@20` 最好 |
| DNNRanker | 0.025540 | 0.891489 | 0.113215 | 0.656796 | AUC 较高，但 TopK 排序仍需优化 |
| TwoTower+DNN-Rerank@50 | 0.025497 | 0.888652 | 0.113354 | 0.656796 | 当前重排未超过单独 Two-Tower |

本轮最重要的结论是：`Two-Tower` 在 `NDCG@20` 上最好，说明短视频推荐中用户兴趣向量和视频内容向量的召回能力已经强于纯热度、类别和 ItemCF。`DNNRanker` 的 AUC 不低，但 TopK 指标没有超过 Two-Tower，说明当前 Ranker 的特征或训练目标还需要针对最终 TopK 排序继续优化。

## 12. 第二轮为什么要做消融实验

第一轮已经证明 `Two-Tower` 在 `small_matrix.csv` 上效果最好，`NDCG@20=0.143577`。
这时继续盲目加模型价值不大，应该开始问更工程化的问题：

1. **标签怎么定义更合理？**
   - `watch_ratio >= 1.0` 表示严格完播。
   - `watch_ratio >= 0.8` 表示接近完播。
   - 如果接近完播效果更好，说明严格完播标签可能太稀疏。

2. **训练数据变多是否真的有帮助？**
   - 第一轮神经模型只采样 120 万训练样本。
   - 第二轮把神经模型扩展到 `small_matrix.csv` 全量训练交互。
   - 目标是判断模型是否吃得下更多数据，以及指标是否提升。

3. **换到更大的 `big_matrix.csv` 后是否还能稳定？**
   - `big_matrix.csv` 有 1,253 万交互，比 `small_matrix.csv` 更接近大规模推荐。
   - 如果模型在大数据上 TopK 下降，就说明现有训练方式还不够强。

4. **Ranker 能不能让重排超过 Two-Tower？**
   - Two-Tower 负责召回候选。
   - DNNRanker 负责给候选重新排序。
   - 如果重排超过双塔，说明 pipeline 有真实收益。

## 13. 第二轮实验结果

### 13.1 标签阈值消融

本实验把正反馈改成：

$$
positive = \mathbb{1}(watch\_ratio \ge 0.8)
$$

核心结果：

| 模型 | Recall@20 | NDCG@20 | AUC | 小白解释 |
|---|---:|---:|---:|---|
| Popularity | 0.007644 | 0.053773 | 0.479285 | 只推热门视频 |
| Category | 0.012311 | 0.085090 | 0.539413 | 按用户喜欢的类别推视频 |
| ItemCF | 0.013332 | 0.104011 | 0.476898 | 根据共同看过的视频找相似视频 |
| MF | 0.018616 | 0.136451 | 0.678856 | 学习用户和视频 embedding |
| Two-Tower | 0.017476 | 0.153744 | 0.513206 | 当前排序位置最好 |
| DNNRanker | 0.018240 | 0.142891 | 0.647959 | 判断正负样本能力强，但 TopK 不如双塔 |

结论：`watch_ratio >= 0.8` 下 Two-Tower 的 `NDCG@20=0.153744`，高于第一轮严格完播的
`0.143577`。这说明“接近完播”更适合作为当前 TopK 推荐标签。

### 13.2 全量 small 训练

本实验保持标签为 `watch_ratio >= 1.0`，但把神经训练样本从 120 万扩展到
`small_matrix.csv` 全量训练交互 `3,595,097`。

| 模型 | Recall@20 | NDCG@20 | AUC | 小白解释 |
|---|---:|---:|---:|---|
| MF | 0.027138 | 0.139644 | 0.691779 | 更多训练数据让 embedding 更稳 |
| Two-Tower | 0.027291 | 0.149288 | 0.535261 | 当前 NDCG 最好 |
| DNNRanker | 0.028359 | 0.146904 | 0.646642 | Recall 更高，但排序位置略差 |

结论：扩大训练数据后，Two-Tower 的 `NDCG@20` 从第一轮 `0.143577` 提升到 `0.149288`。
这说明当前双塔模型能够从更多训练交互中受益。

### 13.3 big_matrix 采样放大

本实验切到 `big_matrix.csv`，使用 200 万神经训练样本。

| 模型 | Recall@20 | NDCG@20 | AUC | 小白解释 |
|---|---:|---:|---:|---|
| Popularity | 0.002464 | 0.008112 | 0.597722 | 大数据上热门推荐很弱 |
| Category | 0.002795 | 0.008871 | 0.606614 | 类别有一点提升 |
| ItemCF | 0.022084 | 0.058148 | 0.605107 | 当前 TopK 最强 |
| MF | 0.001905 | 0.004222 | 0.776112 | AUC 高，但推荐列表命中少 |
| Two-Tower | 0.000682 | 0.001448 | 0.736317 | 当前召回 TopK 不够好 |
| DNNRanker | 0.002511 | 0.004502 | 0.764070 | 排序判断能力强，但全量推荐弱 |

结论：`big_matrix.csv` 上出现了“AUC 高但 TopK 弱”的现象。小白可以这样理解：

- `AUC` 高：模型在抽样正负样本里，能大致判断哪个更像正样本。
- `TopK` 弱：真正从一万多个视频里挑 Top20 时，模型还不能把用户未来喜欢的视频排到最前面。

这说明后续不能只看 AUC，还要重点优化召回训练和 TopK 指标。

## 14. 为什么当前 Ranker 还没超过 Two-Tower

当前 DNNRanker 没有稳定超过 Two-Tower，不代表 Ranker 没用，而是说明第一版 Ranker 还比较弱。

主要原因：

1. Ranker 训练的是二分类任务，主要优化 `AUC` 和 `LogLoss`，不是直接优化 `NDCG@20`。
2. Ranker 只能重排 Two-Tower 召回出来的候选，如果候选里没有好视频，Ranker 也排不出来。
3. caption 当前只是哈希 embedding，没有真正的语义理解。
4. 负样本还不够难，模型没有充分学习“看起来像正样本但用户其实不喜欢”的视频。

后续优化方向：

1. 增强 Ranker 特征：用户历史类别偏好、视频完播率、视频热度、时长、caption 文本。
2. 增强负采样：in-batch negative、hard negative。
3. 尝试 pairwise/listwise 排序 loss，让训练目标更接近 `NDCG@20`。
4. 在 `big_matrix.csv` 上做更完整的神经训练，而不只采样 200 万行。

## 15. 第三轮 Ranker 优化结果

第三轮已经按第二轮的分析做了 hard negative 优化，并取得明确收益。

### 15.1 hard negative 是什么

普通负样本是用户没有完播、没有接近完播的视频。hard negative 是更难的负样本：

> Two-Tower 给它打了高分，觉得它可能适合推荐，但真实标签是负样本。

这类样本非常适合训练 Ranker，因为 Ranker 的任务就是在召回候选里把“看起来像正样本但实际不是”的内容压下去。

### 15.2 本轮设置

| 项目 | 数值 |
|---|---:|
| 标签 | `watch_ratio >= 0.8` |
| 基础训练样本 | 1,200,000 |
| hard negative pool | 1,500,000 |
| 每用户 hard negatives | 100 |
| 追加 hard negatives | 141,100 |
| Ranker 最终训练样本 | 1,341,100 |
| Ranker 正样本权重 | 4 |
| Ranker hidden dims | `256,128,64` |

### 15.3 本轮结果

| 模型 | Recall@20 | NDCG@20 | AUC | 小白解释 |
|---|---:|---:|---:|---|
| Two-Tower | 0.018152 | 0.159630 | 0.508734 | 召回底座 |
| DNNRanker | 0.028238 | 0.240050 | 0.653649 | 直接对全量视频打分，当前 TopK 最强 |
| TwoTower+DNN-Rerank@50 | 0.019543 | 0.170111 | 0.690666 | 小候选集重排已超过双塔 |
| TwoTower+DNN-Rerank@100 | 0.022158 | 0.197553 | 0.654515 | 候选扩大后继续提升 |
| TwoTower+DNN-Rerank@200 | 0.022686 | 0.203215 | 0.687100 | 当前最佳两阶段 pipeline |

结论：第三轮已经解决“Ranker 重排没有超过 Two-Tower”的问题。
`TwoTower+DNN-Rerank@200` 的 `NDCG@20=0.203215`，明显高于单独 Two-Tower 的 `0.159630`。

下一步应把 hard negative 方案迁移到 `big_matrix.csv`，并进一步优化 Two-Tower 召回训练。
